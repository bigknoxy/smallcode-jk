The harness, not the model size.

smallcode is an agentic coding harness engineered to make small local models produce real, correct, shippable code. Target model: VibeThinker-3B (Q8_0) via Ollama — fully offline. The thesis: harness design (tool-exec loop, full-file edit format, deterministic test-oracle early-stop, best-of-N) unlocks usable coding without cloud APIs or frontier models.

How it works

Four structural choices do the heavy lifting. VibeThinker-3B raw single-shot HumanEval pass@1 is ~0.45–0.60. The agentic loop lifts the same model to 0.828 pass@1 / 0.906 solve-rate on the full TypeScript HumanEval.

1

Plan → Goals

The harness calls the model to decompose the task into ordered sub-goals (max 8). Each goal is attempted independently, so failures are local.

2

Tool-exec turns

Each turn the model emits edits and tool calls. The harness actually executes them — read_file, write_file, run_command, run_tests — and feeds real output back. No blind flying.

3

Full-file edit format

Model writes FILE: path + a fenced block with the entire corrected file. No diffs, no SEARCH/REPLACE markers. The key fix: VibeThinker burned its whole budget on marker syntax; full-file jumped the suite from 0.40 → 0.96.

4

Deterministic test-oracle early-stop

After every turn the harness auto-runs bun test. Tests green = task solved. Break immediately to lock the pass, prevent regressions from later turns, and stop burning budget.

Quick start

# 1. Install dependencies bun install # 2. Start Ollama with VibeThinker-3B ollama pull vibethinker-3b ollama serve # 3. Run the first 30 HumanEval problems, k=3 trials each SMALLCODE_HE_LIMIT=30 SMALLCODE_HE_K=3 bun scripts/run-humaneval.ts # 4. Run the full in-house capability suite (dry-run validates fixtures) SMALLCODE_DRY_RUN=1 bun scripts/run-baseline.ts bun scripts/run-baseline.ts # 5. Debug a single task end-to-end bun scripts/debug-task.ts <task-id>

Config is in smallcode.config.json. The default points at http://localhost:11434/v1 (Ollama) with model vibethinker-3b. See LLM Reference for all env vars and config options.

Key design decisions

Why full-file edits instead of SEARCH/REPLACE?

VibeThinker-3B burned its entire context budget agonizing over exact marker syntax (bracket counts, the === separator, whitespace). It emitted nothing parseable. Switching to whole-file rewrite — just FILE: path + a fenced block — is unambiguous. The suite jumped from 0.40 to 0.96 in one change.

Why test-oracle early-stop?

Tests green = task solved with certainty. Breaking immediately locks the pass: subsequent turns cannot accidentally regress a correct solution, and the budget stops being spent. The oracle also handles "should-not-edit" tasks — if tests start green, the loop exits on turn 1.

Why temperature ≥0.6? Never 0.2?

VibeThinker-3B was not trained for low-temperature exploration. At temp 0.2 it degenerates into repetitive, low-diversity output that fails to self-correct. Required sampling: temp 1.0 (or minimum 0.6), top_p 0.95. High variance is addressed by best-of-N and the k=3 eval protocol.
