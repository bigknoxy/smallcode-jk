# smallcode harness roadmap — awesome-harness-engineering research pass

## Executive summary

Read `ai-boost/awesome-harness-engineering` end-to-end (~30 categories, 566-line README) and followed 8 high-signal sources (outlines, statewright, TraceCoder, llvm-autofix, AgentAssay, AgentLens, LaMR context-pruning). Most of the list is enterprise/cloud/frontier-model tooling (permissions, observability, human-in-the-loop, production infra) that doesn't transfer to a fully-offline sub-14B local-model TDD bug-fixer — excluded per the skepticism filter. Three items clear the bar as genuinely new and cheap to test: **state-gated tool access** (statewright — directly explains *why* smallcode's target-lock already works, and generalizes to a 3-phase gate), **process-quality scoring on top of binary solved/not-solved** (AgentLens's "lucky pass" taxonomy — cheap, reuses existing transcripts, no model calls), and **constrained decoding for edit-format validity** (outlines — high payoff but blocked today by Ollama's HTTP API not exposing logits, so it's P2/parked pending a llama.cpp-direct serving path). Two items were investigated and explicitly rejected for smallcode's shape: LaMR's AST/CRF retrieval (infra-heavy, marginal over an already-40/40 lexical scorer) and TraceCoder's HLLM (mechanism unspecified in the abstract, redundant with existing forensic-task workflow). AgentAssay's behavioral fingerprinting is noted but deprioritized — it targets *tool-use* drift in interactive agents, and smallcode's loop is closed-form enough that pass@k CIs already catch what fingerprinting would.

## P0 — do next

### 1. Lucky-pass / process-quality scoring on existing eval transcripts — IMPLEMENTED
- **Source**: AgentLens, https://arxiv.org/abs/2605.12925
- **Technique**: classify a *passing* trajectory as Lucky (blind retry, missing verification, disordered explore/implement/verify) vs Solid vs Ideal, using intent-labeled action sequences — not a new metric, a re-read of trajectories already logged.
- **Why it fits**: smallcode already logs full turn-by-turn transcripts (`src/eval/transcript-store.ts`, `scripts/show-transcript.ts`, `scripts/triage-transcripts.ts`) and scores binary "solved" via the tiered oracle (`src/verify/oracle.ts`). AgentLens found 10.7% of SWE-bench passes were lucky (up to 23.2% for some models) — exactly the failure mode the project's own culture ("forensic > aggregate") already suspects but has no cheap detector for. Zero new model calls, zero new local-model burden — it's a static-analysis pass over transcripts already on disk.
- **Implementation**: `scripts/classify-pass-quality.ts` + `tests/classify-pass-quality.test.ts`, reading `transcript-store` output. Pure `classifyPassQuality()` flags a solved run as Lucky if (a) it reverted-and-retried ≥2 times before green with no diagnostic content change between attempts (churn), (b) the run struggled (revert/many-turns/multi-signature) and never carried a diagnostic (never-localized), or (c) the solving edit itself carried no diagnostic though earlier turns did (untargeted-fix). Clean few-turn explore→edit→verify → Ideal; everything else → Solid.
- **Model-solved vs harness-rescued split (added after operator-mutation repair shipped, see item below)**: a passing transcript is now attributed FIRST — before any lucky/solid/ideal logic runs — as `rescued` iff any turn carries `TurnRecord.mutationRepair` (the harness's deterministic operator-mutation brute-force solved it, the model's own turns all failed). Rescued runs are mutually exclusive with lucky/solid/ideal and excluded from the lucky-rate denominator (rescued isn't a model trajectory to grade). The CLI/JSON now report `rescued` counts plus **model-solve rate** = (ideal+solid+lucky)/total-passing and **rescued rate** = rescued/total-passing, alongside the existing lucky-rate (now computed over model-solved runs only, not diluted by rescues).
- **Payoff + measurement**: run against the existing realrepo/edit-reliability suites' stored transcripts (no new eval run needed). Report lucky-rate per task alongside pass@k. If lucky-rate correlates with tasks already flagged as near-floor (mri, dequal-ish zeros), that's a free diagnostic; if BoN's "resolves on first green" wins are mostly Solid not Lucky, that's a validity confirmation for the BoN result already banked in memory. Purely additive — can't regress anything since it doesn't touch the agent loop. The rescued split additionally lets pass@1 be read honestly against the operator-mutation-repair win (PR #108) — a task's headline "solved" number can now be decomposed into what the model actually did vs what the harness's last-resort brute force did.

### 2. Name statewright's mechanism explicitly and extend target-lock to a 3-phase gate
- **Source**: statewright, https://github.com/statewright/statewright
- **Technique**: per-workflow-phase tool-schema restriction (planning phase → read-only tools; implementation phase → edit unlocks; separate testing phase) — reported 2/10→10/10 on a 5-task local-model SWE-bench subset, attributed explicitly to "breaking read-loop death spirals and keeping the tool space small enough that the model reasons instead of flailing," not model size.
- **Why it fits**: this is the *mechanism name* for something smallcode already partially does — target-lock + retarget-on-mispin (per project memory) already constrains *which file* the model can drift into. statewright generalizes the same idea to *which tools* are visible at all, per phase. smallcode's tool surface (`src/agent/tools.ts`) is already small relative to a general coding agent, so the marginal win is smaller than statewright's headline number — but the specific gap is real: nothing currently stops the model from attempting an edit-tool call before retrieval/planning has locked a target, which is exactly the "sequencing error" statewright's 3-state minimum targets.
- **Implementation sketch**: in `src/agent/loop.ts` / `src/agent/strategy.ts`, gate tool availability by turn state: before target-lock, only read/search tools are exposed to the prompt; after lock, edit tools unlock; during oracle-verify, no mutating tools. This is a prompt-construction change (which tool schemas get serialized into the turn), not a new subsystem.
- **Payoff + measurement**: A/B on the existing edit-reliability + realrepo suites, same protocol as prior BoN/line-edit A/Bs (pooled pass@1 + per-task bootstrap CI, forensic check on any task that moves). Cheap to falsify — if smallcode's tool surface is already small enough that this is a no-op, that's a valid, useful negative result consistent with "already have" (worth banking either way in project memory).

### 3. `lru-recency` floor — CRACKED 2026-07-03 (RAD_HINT + STATEMENT_REPAIR; BoN lever also confirmed)
- **Source**: internal — the identified hard floor of the realrepo hard-tier expansion (PR #112, suite 14→22). This is the "de-saturate the stick then chase the new floor" step.
- **The wall**: `realrepo-lru-recency_1` sits at **~0.09** for qwen-7b (n=8=0.00, n=3=0.33). Saved transcripts (`evals/transcripts/realrepo-lru-recency_1/`) show the model **localizes perfectly** every turn (single-file, always edits `src/index.js:get`) and writes the **right shape** (delete + re-set for recency) but with a consistent **read-after-delete ordering error** — `map.delete(key); map.set(key, map.get(key))` re-inserts `undefined` — and stays stuck in that local minimum to the turn cap. So the floor is NOT a localization gap and NOT operator-mutation-repairable (no operator token in the buggy `return map.get(key)` body). It is a narrow multi-line reasoning error (wrong statement order / needs a temp binding before the delete).
- **Confirmed lever — Best-of-N (already shipped, no new code)**: `SMALLCODE_BEST_OF_N=5` (7b, n=8, mutation-repair off) lifted it **0.00 → 0.38** [.00-.75] (0/8 → 3/8), matching the predicted `1-(0.9)^5 ≈ 0.41` from a ~0.1 per-attempt right-shape base rate. Diversity resampling lands the correct order the greedy pass misses. Reproduced 0.38 [.13-.75] (CI-significant, lower bound > 0) on a second run. Contrast `mri` ([[project_mri_escalation_ceiling]] in memory): BoN could NOT move mri (a genuine idiom-comprehension zero) but DOES move lru-recency — BoN lifts a floor the model can *sometimes* clear, never a true zero.
- **CAVEAT — the BoN lift is all luck, not comprehension (lucky-pass audit, `scripts/classify-pass-quality.ts`)**: every BoN lru-recency solve classifies **lucky** (100% lucky-rate) — blind resampling until an attempt happens to land the right statement order, no targeted diagnosis. BoN buys pass-rate, NOT model capability; it is a brute-force diversity crutch in the same category as operator-mutation repair (harness-assisted, not a model gain). So the recommendation is tempered: BoN is a legitimate *pass-rate* lever for hard multi-line-structural tasks, but moving actual model comprehension needs a different lever (a deterministic statement-repair, or in-context feedback surfacing the read-after-delete failure). The same audit confirms the harness-rescue split works: `mri` = 5/5 rescued (0% model-solve), `modwrap`/`nullmerge` = genuine model solves, `csv-quote` = 60% lucky (fragile multi-file localization passes).
- **SHIPPED — two opt-in levers CRACK the floor 0.13 → 1.00 (2026-07-03), one from each side of the model-solved/harness-rescued split**. New pure module `src/repair/read-after-delete.ts` (no AST, no deps, mirrors `src/repair/operator-mutation.ts`): `detectReadAfterDelete(source)` conservatively scans for the `X.delete(K); X.set(K, X.get(K))` pattern + returns a human hint; `repairReadAfterDelete(source)` deterministically hoists the read into a `__radVal` temp before the delete. Both read the model's most-recent attempt via the new exported loop helper `latestAttemptContent(state, targetPath)` (mirror of `pristineTargetContent`) — required because the read-after-delete attempt stores `undefined`, regresses green tests, and is reverted off disk, so disk is pristine.
  - **`SMALLCODE_RAD_HINT` (model-side, default ON since v1.7.1) — REAL MODEL GAIN, the first model-side lever to move a genuine realrepo floor**: when a failing turn leaves the pattern in the attempt, the next prompt carries a `## STATEMENT ORDER BUG` section telling the model to read the value into a variable BEFORE the delete; the MODEL reorders and solves. A/B (7b, n=8, mutation-repair off): pass@1 **0.13 [.00-.38] → 1.00 [1.00-1.00]** (1/8 → 8/8), avg turns **5.0 → 2.3**, lucky-pass audit **modelSolveRate 1.0, rescued 0**. Causal mechanism confirmed in transcripts: T1 model writes the bug → T2 prompt carries the hint → model reorders → passes. Contrast `mri` ([[project_mri_escalation_ceiling]]): NO model-side lever ever moved mri (idiom-incomprehension) — lru-recency is a comprehension bug the model CAN fix once told the exact ordering error. **Promoted to default-on (v1.7.1, 2026-07-04)** after a full 22-task realrepo A/B (7b, n=8, control=defaults vs treatment=RAD_HINT+STATEMENT_REPAIR on): the detector fired on the `lru-recency` read-after-delete pattern **only** — 0 fires across the other 21 tasks / 176 trials — so it is inert everywhere else. The A/B surfaced two other-task swings (`toposort-cycle` 1.00→0.63 CI-flagged, `csv-quote` 0.63→0.13) but a lever-off control rerun redrew `toposort-cycle` 0.88 and `csv-quote` 0.50, confirming both are sampling noise on lever-untouched tasks, not lever-caused. `STATEMENT_REPAIR` stays opt-in (it never fired in the full suite once the hint solved `lru-recency` first).
  - **`SMALLCODE_STATEMENT_REPAIR` (harness-side, default off) — harness rescue** (the deterministic sibling foreseen below, now built as `read-after-delete.ts` not `statement-mutation.ts`): after an unsolved fix-mode run with a locked target, recovers the (possibly reverted) attempt, hoists the read into a temp, runs the real oracle, keeps first green. A/B (7b, n=8): pass@1 **1.00 [1.00-1.00]** (8/8), every solve logged `[statement-repair] SOLVED src/index.js via read-after-delete hoist`, recorded as `mutationRepair` so the lucky-pass audit attributes it **rescuedRate 1.0, model 0**. Disjoint bug-shape from operator-mutation repair (no operator flip fixes an ordering mistake). Full suite 1039 tests green, tsc clean.

### 4. Final-state regression guard (dogfood Gap 2 "never leave the repo worse than found") — SHIPPED 2026-07-04 (opt-in), PROMOTED to default-on 2026-07-05
- **The gap** (from the real-`run` dogfood audit + external agent-harness research, Hermes/Roo/Cursor all ship shadow-git checkpoint + end-state rollback): the per-turn revert catches regressions *during* the loop, but (a) it cannot roll back a brand-new file the model created (no pre-edit `originalContent` to restore — the apply layer records `undefined`), and (b) it never re-checks the whole-run end state. So a wandering / partial run could exit with more red on disk than it started — exactly the "left the repo BROKEN" dogfood failure. `run.ts`'s `classifyCompletion` only reads `status × verified`; it never re-runs the oracle vs baseline.
- **SHIPPED — `SMALLCODE_FINAL_STATE_GUARD` (harness-side, default ON since 2026-07-05, set `=0` to disable)**: runs absolutely last, only when the run ended UNSOLVED after the model loop AND every repair pass. New pure oracle decision `finalStateWorseThanBaseline(baseline, final)` (in `src/verify/oracle.ts`) + new exported loop helper `pristineRunSnapshot(state)` (whole-run generalization of `pristineTargetContent` — first-seen `originalContent` per touched path, plus the set of brand-new files created) + I/O gate `runFinalStateGuard` (in `src/agent/loop.ts`). Recaptures the full `bun test` baseline on the final disk state; if strictly worse (higher red count OR a test failing now that was green/absent at baseline), restores every touched file to pristine (`revertFiles`) and deletes created files, then re-verifies the restore reached ≤ baseline (logged `[final-state-guard] reverted N file(s) …`, recorded on `state.finalStateReverted`).
- **Eval-neutral by construction**: fires only on unsolved runs and reverts to the seeded-bug START state, so an unsolved trial stays unsolved (pass/fail unchanged) — it removes broken residue, it never manufactures a pass. Partial progress (fewer/same failures, no NEW failing test) is explicitly NOT reverted. 11 new tests (pure decision + snapshot + real-oracle mechanism + solved-run eval-neutrality), full suite green, tsc clean.
- **Why opt-in first**: same discipline as RAD_HINT/STATEMENT_REPAIR — ship the mechanism default-off, validate on the real dogfood repos (does it actually catch broken-exit without harming partial-progress runs on flaky suites?), then promote to default-on.
- **PROMOTED to default-on 2026-07-05**: validated eval-neutral — realrepo subset pooled pass@1 0.90 = baseline (unchanged), confirming the guard fires only on already-unsolved runs and never costs a pass. `env.finalStateGuard` now defaults `true`; set `SMALLCODE_FINAL_STATE_GUARD=0` to opt back out.
- **FIXED — dogfood #3 gap, 2026-07-08 (throw containment)**: a real `smallcode run --escalation` run left a broken edit on disk (suite 2→6 red) with the guard NOT firing (no `[final-state-guard]` log, `finalStateReverted` stayed `null`). Root cause: `runOperatorMutationRepair`/`runStatementRepair` run the real `bun test` oracle per candidate, which can THROW (e.g. a timeout on a large repo); the throw was unguarded and escaped `runLoop` entirely, orphaning a half-tried candidate on disk AND skipping the guard that runs right after (the guard logic itself was correct in isolation — it was simply never reached). Fix: both repair fns wrap their candidate loop in try/catch — on any throw, restore the file to the model's pre-repair edit and return `null` (UNSOLVED) instead of propagating (logged `[mutation-repair] … aborted …` / `[statement-repair] … aborted …`); `runLoop` additionally wraps both repair-pass blocks in one outer try/catch as defense-in-depth. Both repair fns gained an injectable `runOracle` param (test-only seam, defaults to `runTieredOracle`) so the throw path is covered deterministically without a global module mock. New tests: `tests/loop-repair-throw-restore.test.ts`, `tests/loop-guard-mutation-repair-repro.test.ts`. `runOperatorMutationRepair`, `runStatementRepair`, `runFinalStateGuard` now exported from `src/agent/loop.ts`/`src/agent/index.ts`. Eval-neutral (only changes throw-path behavior).

### 5. Static import-resolution gate (dogfood Gap 4 "hallucinated APIs / imports") — SHIPPED 2026-07-04 (opt-in), PROMOTED to default-on 2026-07-05
- **The gap** (from the dogfood audit + external research — the #1-ranked technique across SWE-agent/OpenHands/Aider/agent-lsp writeups is grounding edits in symbols that actually exist): the model invents a module (`std/strings`) and never recovers, because Gap 1's R4 feedback is REACTIVE — it surfaces `Cannot find module` only after a full test run, and only when a test imports the edited file, so the model loops re-emitting the same invented import. Grounding is a capability failure a 3B can't self-diagnose; the harness must resolve "does this exist?" deterministically.
- **SHIPPED — `SMALLCODE_IMPORT_GATE` (harness-side, default ON since 2026-07-05, set `=0` to disable)**: new pure-extraction + fs-resolution module `src/verify/import-check.ts` (`extractImportSpecifiers` / `diffNewSpecifiers` / `resolveSpecifier` / `checkNewImports` / `formatImportRejection`). Wired in `src/agent/loop.ts` post-apply, pre-oracle: for each `FILE:`/`PATCH:` edit that landed on an existing source file, resolve the specifiers the edit INTRODUCED — relative via `Bun.resolveSync` from the file dir, bare against `package.json` deps + `node_modules`, builtins OK — and any that don't resolve revert the file to pre-edit content + feed a targeted `IMPORT ERROR — … does not resolve … this repo's dependencies are: …` naming the real deps. Crisper/earlier/more-actionable than R4's post-test-run signal.
- **Conservative by design** (near-zero false-rejects): a declared dep or any Bun-resolvable specifier passes; only NEW imports on an existing file are checked; test/brand-new files skipped. 18 new tests (pure extractor/classify/diff, fs-backed resolver, real-repo `checkNewImports`, runLoop integration), full suite green, tsc clean.
- **Why opt-in first**: same ship-discipline — validate on real dogfood repos, then promote. `write_file`-path imports are still covered reactively by R4 (this gate covers the primary `FILE:`/`PATCH:` path the dogfood failure used); extending to the `write_file` path is a follow-up if validation warrants.
- **PROMOTED to default-on 2026-07-05**: validated live — the gate fired and reverted a real 3B hallucinated `import ... from "slugify"`, with zero false-fires on an import-prone task; the accompanying resolver fix (bare packages resolve against THIS repo's declared deps, not Bun's global install cache) landed on the same branch. `env.importGate` now defaults `true`; set `SMALLCODE_IMPORT_GATE=0` to opt back out.

### 4. AgentAssay-style behavioral fingerprinting for regression-gating (`scripts/regression-gate.ts`)
- **Source**: AgentAssay, https://arxiv.org/abs/2603.02601
- **Technique**: map execution traces to compact behavioral vectors; use a stochastic PASS/FAIL/INCONCLUSIVE verdict (hypothesis test across multiple trials) instead of binary single-run regression checks. Reported 86% regression-detection power vs 0% for naive binary testing, at 78-100% lower token cost than re-running full evals.
- **Why it fits, cautiously**: smallcode already has `src/improve/regression-gate.ts` and pass@k + bootstrap CIs (per project memory, "Eval measuring stick") — this is the one item where the existing infra is closest to redundant with the proposed technique. The genuine addition is orthogonal, not a replacement: CIs answer "did the success rate change," fingerprinting would answer "did the model's *behavior* change even at equal success rate" (e.g. a prompt tweak that starts winning via more BoN retries instead of first-shot correctness — same pass@1, worse cost profile). That's a real blind spot pass@k alone doesn't catch.
- **Implementation sketch**: extend `regression-gate.ts` to also diff a cheap behavioral signature per task — e.g. (turns-to-solve, BoN-attempt-index-of-first-green, edit-format used, revert count) as a vector — and flag drift even when pass@k is CI-unchanged.
- **Payoff + measurement**: retrofit onto historical A/B pairs already in project memory (e.g. the BoN win, the line-edit rejection) as a validation check: does the fingerprint diff correctly distinguish the confirmed-real BoN win from the confirmed-noise think-only/temp experiments? If yes, ship as a gate; if the vector doesn't discriminate, that's a clean negative result — don't force it.

## P2 — parked / needs a blocking dependency resolved

### 5. Constrained decoding (grammar-locked edit format) via outlines
- **Source**: outlines, https://github.com/dottxt-ai/outlines
- **Technique**: regex/CFG/JSON-schema-constrained token sampling — filters the model's next-token distribution so only grammar-valid continuations are ever sampled. Directly targets the exact failure class smallcode already spends real effort on (elision detection, forced minimal-PATCH, size-gating, `src/edit/repair.ts`) — malformed edit blocks.
- **Why it's promising**: this is the most surgical possible fix for edit-*format* reliability (as opposed to edit-*content* correctness, which is the capability ceiling BoN/escalation already address). A grammar-locked SEARCH/REPLACE or unified-diff format would make `src/edit/repair.ts`'s repair-on-malformed-output path structurally unnecessary for a whole class of errors.
- **Blocking dependency (confirmed via fetch)**: outlines requires direct logit access. It supports llama.cpp and transformers natively, and documents Ollama support at the server level, but **standard Ollama HTTP API responses do not expose raw logits** — confirmed as the integration blocker. smallcode's provider layer (`src/provider/openai-client.ts`, `src/provider/types.ts`) currently talks to Ollama over its OpenAI-compatible HTTP endpoint, which is exactly the black-box surface outlines can't constrain.
- **Implementation sketch (parked)**: would require either (a) swapping the provider to hit llama.cpp's server directly (which does support grammar-constrained sampling via GBNF, no outlines dependency needed) or (b) using Ollama's own native `format` JSON-schema parameter (Ollama has shipped structured-output support independent of outlines) scoped to a diff-block schema. Path (b) is more realistic — it's already in the current provider's dependency, not a new one.
- **Payoff + measurement**: if pursued, A/B against the existing edit-reliability suite's malformed/elided-block rate (already tracked, since `repair.ts` exists to fix exactly this). Expected payoff is a reduction in repair-path invocations, not necessarily a pass@1 lift (repair already recovers most cases) — so the real metric is turns-saved / latency, not pass@k. Recommend re-opening this only after confirming whether Ollama's native structured-output mode can express a diff grammar (2-hour spike), rather than committing to the outlines dependency.

#### RESOLVED 2026-07-02 — **NO-GO** (spike done: design memo + live Ollama probe)
- **Ollama exposes constrained decoding ONLY via JSON Schema** (`format` / `response_format`) over HTTP. Raw GBNF is **not** reachable — the PR to expose a `grammar` param ([ollama/ollama#4525](https://github.com/ollama/ollama/pull/4525)) was closed in favor of structured-outputs (#7900). So the marker-grammar path (a) is dead; only the JSON-envelope path (b) is real.
- **Plumbing already exists, unused**: `CompletionRequest.responseFormat` flows to the HTTP body (`src/provider/openai-client.ts:44-73`); zero callers set it. A JSON edit-envelope format already exists in `src/edit/parser.ts:202-293`.
- **Decisive empirical probe** (qwen2.5-coder:7b, local, matched multi-line hard cases): JSON-envelope constrained = **0/6** correct+byte-exact vs free-text SEARCH/REPLACE = **5/6** on the *identical* bugs. Forcing code into a JSON string value makes the model **flatten newlines/whitespace**, so `search` is no longer byte-exact and the edit silently won't apply. The `format` grammar guarantees syntactic JSON validity (100%, negligible latency) but **defeats** the byte-exact multi-line reproduction SEARCH/REPLACE depends on — the same "added-precision → regression on a small model" pattern as the rejected LINE_EDIT experiment.
- **Framing**: constrained decoding fixes *format validity* (which `repair.ts` already recovers), not the search-drift / wrong-line failures that are the real ceiling. **Verdict: keep the free-text SEARCH/REPLACE format.** Structured output stays fine for non-code metadata (classifications, single-line answers) with no internal whitespace.
- **Only cheap next lever if edit-format is revisited**: instrument the repair-path invocation rate first (a counter around `repairBlock` / `strategy !== "exact"` in `src/edit/applier.ts:378-385`) over the edit-reliability suite — no telemetry exists today, so the payoff can't even be sized. Do that *before* any format change.
