src/agent/

FilePurpose
loop.tsCore agent turn loop — planning, model calls, edit application, tool execution, baseline-relative test-oracle early-stop. Captures a TestBaseline before the loop starts; passes it to every runTieredOracle call. Implements stall detection: STALL_LIMIT=2 consecutive turns with the same failureSignature trigger a redraft (suppresses Recent History, injects rotateStrategy hint); capped at MAX_REDRAFTS=2. Think-only recovery: when a turn truncates mid-reasoning (hasReasoning && answer==="" && truncated!==false) it is recorded as a failed turn and answerNowNext is set so the next turn is built with { answerNow: true } (suppress history, demand immediate action) instead of re-sending the identical prompt that just truncated. Empty-generation infra guard: a turn whose provider returned zero tokens (rawContent.trim()==="" && completionTokens===0 — a wedged/disconnected local backend that emits {"response":"","done":false}) is thrown and recorded with the error tag "infra: empty model generation", distinct from think-only (which HAS reasoning), so the eval can exclude the trial instead of forging a false 0.00. Revert-on-regression: after runTieredOracle returns, if verdict.regressed is true (the edit introduced new failures: newFailures are named, OR total red count exceeded the baseline red count — which catches module-load crashes that print no parseable failure names), the loop calls revertFiles using each ApplyResult.effectivePath (the path actually written, after path-typo rescue) and ApplyResult.originalContent (pre-batch on-disk content) to restore file state precisely. The revert set ALSO snapshots every TOOL: write_file target's pre-turn content (captured via readFileFn before tool execution) — that write path bypasses applyBatch, so without it a build-breaking write_file was silently NOT reverted (the dogfood bug: a garbage write_file survived 5 turns to max_turns); now a regression rolls back regardless of which write path produced it. Reverts only on true regression, never when the suite is still-failing because the target isn't fixed yet (progress is kept); "solved" never reverts. On revert the turn's stall signature folds onto the prior turn's so a repeated regress→revert loop still trips the stall/redraft brake. The model receives in the next turn: "⚠ Your edit was REVERTED — it broke tests that were passing before: <names>…". Stores turn.failureSignature, turn.diagnostic, turn.redrafted, turn.answerNow; persists state.stallCount, state.redraftCount, state.lastFailureSignature. Off-task-drift guard: when context.targetFile is a confidently-pinned single edit target AND verdict.outcome==="failing" AND there is a LATER goal to advance onto (currentGoalIndex+1 < goals.length), TOOL: finish no longer calls advanceGoal — the loop stays anchored to the SAME goal (and therefore the same retrieval-resolved target file) instead of sliding onto the planner's own trailing "write tests"/"run tests" sub-goals, which is how the model wandered into unrelated files turn after turn in live dogfood (found via 2 real 7b runs: turn 1 lands a wrong edit on the correct target, turns 2-8 never touch it again). Single-goal tasks (no later goal to drift onto, e.g. each Best-of-N attempt) are unaffected — finish() still ends the run exactly as before. Target-lock (drift ENFORCEMENT, SMALLCODE_TARGET_LOCK, default on): the prompt-level "stay on target" instruction above is advisory only — dogfooding showed a 7b simply ignoring it and editing an unrelated file 7x. This is the hard-reject version: fixModeBaseline is computed once from the pre-loop testBaseline (hadAnyTests && (failingIds.size>0 || redCount>0) — a genuinely red baseline, the drift-prone bug-fix regime). Stable run-level lock (post-#80 fix): a bare per-turn binding (lockTargetPath = context.targetFile.path, recomputed every turn) DRIFTS — context is rebuilt each turn via getContext, so once the model edits an off-target file, that file enters recent-history/context and retrieval re-pins context.targetFile onto it, silently relocating the "lock" onto the drifted file (or to undefined) and rejecting nothing (live dogfood: an unrelated file edited 6x with ZERO rejections). The fix captures the target ONCE: the FIRST turn where state.lockedTargetPath === undefined && fixModeBaseline && context.targetFile !== undefined, state.lockedTargetPath is set to context.targetFile.path and persisted via saveState; it is never overwritten afterward. Every turn's enforcement then reads lockTargetPath = env.targetLock && fixModeBaseline && state.lockedTargetPath !== undefined ? state.lockedTargetPath : undefined — the STABLE value, not the live per-turn context.targetFile — threaded into applyBatch(..., { targetPath: lockTargetPath }) for FILE:/PATCH: blocks, AND checked directly (via isOnTargetPath) before executing any TOOL: write_file call — the second write path that bypasses applyBatch entirely. Either path rejects an off-target write before it touches disk, with feedback naming both the (stable) target and the rejected path; prompt.ts's "## STAY ON TARGET" block also names state.lockedTargetPath when set, so the prompt and the enforcement always agree on which file is "the target". No targetFile ever confidently pinned (multi-file/low-confidence retrieval) or a clean baseline (no red test to fix) never enforces — state.lockedTargetPath stays undefined for the whole run. Retarget on persistent mis-pin: trackOffTargetAttempt tracks state.offTargetStreak, a consecutive same-path counter over rejected off-target writes; 2 consecutive attempts at the SAME non-test source path retargets state.lockedTargetPath to it (and applies that turn's write) instead of rejecting the mis-pinned lock forever — see the SMALLCODE_TARGET_LOCK env-var row for the full mechanism. Read-after-delete levers: latestAttemptContent(state, targetPath) (exported, mirror of pristineTargetContent) recovers the model's most-recent applied content from turn history even after the loop reverted it off disk — required by both SMALLCODE_RAD_HINT (model-side hint) and SMALLCODE_STATEMENT_REPAIR (harness-side deterministic hoist via src/repair/read-after-delete.ts), because the read-after-delete attempt stores undefined, regresses green tests, and is reverted, leaving disk pristine. Final-state regression guard (SMALLCODE_FINAL_STATE_GUARD, default ON since 2026-07-05): runs absolutely last, only when the run ended UNSOLVED after the loop AND every repair pass — runFinalStateGuard recaptures the end-of-run bun test baseline and, if it is strictly worse than the run-start baseline (finalStateWorseThanBaseline), restores every touched file to pristine via the exported pristineRunSnapshot(state) helper (whole-run generalization of pristineTargetContent — first-seen originalContent per path) + revertFiles, and deletes brand-new files the agent created, honoring the "never leave the repo worse than found" guarantee; eval-neutral because it only fires on unsolved runs and reverts to the seeded-bug start state (records state.finalStateReverted). Repair-pass throw containment (dogfood #3, 2026-07-08): runOperatorMutationRepair and runStatementRepair (both exported, along with runFinalStateGuard, and re-exported from src/agent/index.ts) each wrap their per-candidate bun test oracle call in try/catch — a throw (e.g. an oracle timeout on a large repo) restores the file to the model's pre-repair edit and returns null instead of propagating (logged [mutation-repair] … aborted … / [statement-repair] … aborted …); previously an unguarded throw escaped runLoop, orphaning a half-tried candidate on disk and skipping the final-state guard entirely. runLoop also wraps both repair-pass blocks in one outer try/catch as defense-in-depth so the guard is unconditionally reached. Both repair fns take an injectable runOracle param (test-only seam, defaults to runTieredOracle) for deterministic throw-path tests. Entry point: runLoop(state, statePath, deps, getContext)
planner.tsCalls the model to decompose a task into ≤5 ordered sub-goals (down from 8). Falls back to a single goal if the model returns nothing parseable. Optional preSolveReflection (default off): one extra cheap call to restate the problem + key edge case, fed into goal generation (AlphaCodium-lite). Context summary is sanitised to unique file paths only (no line ranges, capped at 8 files) so the model cannot echo path labels back as goals. Post-parse filtering via isActionableGoal(desc) rejects bare path / path-with-line-range echoes, and isInstructionEcho(desc) rejects goals that parrot the planner's OWN instruction prose (failure mode #7 — markers like "action verb", "sub-goal", "numbered list", literal <action verb>/<specific thing> placeholders, or the copied example verb list "Add, Fix, Implement"); when every goal is filtered out the planner falls back to the raw task. maxGoalsForTask(task, uniqueFileCount) enforces the cap (single-file → 3, multi-file → 5). Exact-duplicate descriptions are deduplicated. The planner system prompt also carries an explicit "NEVER restate these instructions as a goal" guard. All three helpers are exported for testing.
prompt.tsbuildSystemPrompt — delegates to config.promptSet ?? defaultPromptSet(); when the resolved PromptSet.skill is non-empty, appends \n\n## SKILL\n<skill> to the returned string; otherwise output is byte-identical to the no-skill baseline. The discipline-rules toggle relocates to prompt-set.ts. buildTurnPrompt(state, context, opts?) — assembles task, goal, turn number, last 2 turns of history (suppressed on opts.redraft), context chunks, a ## Edit Target directive derived from context.targetFile (small file ≤80 lines → "emit the COMPLETE file" via FILE:; large file >80 lines with a known function → "use PATCH:/FUNCTION: to replace ONLY that one function" — the harness, not the model, picks the format), and a Failure (fix THIS) block rendered from state.turns.at(-1)?.diagnostic when present. When the edit target is an anonymous export default function (…) (synthetic anchor name "default"), the directive prose references the recognizable export default function rather than the opaque "the default function" (which names nothing visible in the source and cost localization confidence on real-repo default exports like mri/klona/dequal); the PATCH FUNCTION: line keeps default as the applier anchor. Format-consistent not-applied feedback: when a SEARCH/REPLACE edit fails to apply (DIFF_EDIT mode, target function ≥ SMALLCODE_DIFF_MIN_FN lines), the retry message instructs the model to re-emit a SEARCH/REPLACE block with the SEARCH text copied byte-for-byte from the shown current file — not "re-emit the complete file", which contradicts the SR directive and causes truncation on large functions. Whole-file and small-function PATCH targets keep their existing not-applied feedback unchanged. When an edit is rejected by the anti-fake-green guard (error contains TEST_FILE_EDIT_REJECTED), the feedback pivots the model to the IMPLEMENTATION file and suppresses the generic "re-emit the file" recovery (which would contradict the rejection and re-show the test). Whole-file-vs-PATCH mismatch recovery (checked FIRST, ahead of the SR/PATCH retry branches above): correlates turn.editBlocks[i] with turn.applyResults[i] (same index, one result per input block) to detect a PATCH-target file where the model instead answered with a whole-file-shaped block (search === "") or hit the ELISION_DETECTED guard — the real-repo dogfood failure where a model told to PATCH one function re-emits an abbreviated whole file with // ... placeholders, gets rejected, and repeats the identical mistake every turn (max_turns, zero progress). Re-showing the whole file and saying "don't emit the whole file" (the old fallback) reinforces exactly this pattern, so instead the feedback calls extractFunctionSource to isolate ONLY the target function's current text and forces a literal copy-pasteable <<<<<<< SEARCH / ======= / >>>>>>> REPLACE template with explicit "Do NOT re-emit the file. Do NOT use // ...." wording, escalating to sterner "FINAL chance" wording via a wholeFileMismatchSeen counter if the same mismatch recurs in the last 2 turns. Default-on, no env gate. Off-task-drift re-anchor ("## STAY ON TARGET", rendered inside the failure block whenever context.targetFile is set): names state.lockedTargetPath ?? context.targetFile.path — the STABLE run-level lock when one has been established, not the live per-turn context.targetFile, so the prompt always names the same file loop.ts's enforcement actually rejects edits against, even if retrieval has drifted onto a different file this turn. Repeats "Edit ONLY <target>. The failing test is <assertionId>…" every turn the target's test stays red; if the exported helper turnEditedPaths(turn) (union of applied edit-block paths, preferring effectivePath, and write_file TOOL-call paths) shows the PRIOR turn wrote to some OTHER file, escalates to a named warning ("⚠ You edited <other file> last turn, but the target is <target>…") instead of the plain restatement. Pairs with loop.ts's off-task-drift guard, which keeps the goal (and thus the retrieval target) from changing underneath this re-anchor. opts.answerNow emits an ANSWER NOW section (skip thinking, act on the first line) and also suppresses history; it takes precedence over redraft. BuildTurnPromptOpts: { redraft?: boolean; strategyHint?: string; answerNow?: boolean }. fitTurnPromptToWindow(state, context, systemPrompt, hardCap, opts?) — pre-flight guard: rebuilds the prompt while shedding the largest non-pinned repo-context chunk each pass until estimateTokens(system) + estimateTokens(prompt) ≤ hardCap or no droppable chunks remain; returns { turnPrompt, estimatedTokens, droppedChunks }. The pinned target file is never shed. History (failing-test output, failed-edit file) is never trimmed. Pure, deterministic, does not mutate the caller's chunk array.
prompt-set.tsPromptSet { system, planner, reflection, skill? } — groups all three agent prompts so a GEPA candidate can carry a mutated variant. The optional skill?: string slot holds a repo-scoped guidance block (mirrors gskill's SKILL.md); when non-empty, buildSystemPrompt appends a ## SKILL section after the base system text. defaultPromptSet(opts?) assembles the canonical set and accepts opts.skill (default undefined). Single source of truth for the prompt strings.
state.tsCreate, load, save (atomic rename), and mutate AgentState. addTurn, advanceGoal, failGoal, isTerminal, currentGoal.
journal.tsWrite-ahead apply journal / crash recovery (SMALLCODE_APPLY_JOURNAL, default ON). The per-turn revert and final-state guard only fire if the loop REACHES them — a process kill / OOM / backend disconnect mid-apply skips both, leaving a half-written repo with no rollback. Before each on-disk edit, recordOriginals(repoRoot, runId, startedAt, paths, capture) persists that file's PRE-RUN content (or a "did-not-exist" marker) to a journal OUTSIDE the repo — journalPathFor(repoRoot) = os.tmpdir()/smallcode-journal/<sha256(repoRoot)[:16]>.json, keyed per-repo so eval trials in distinct throwaway dirs never collide. First-seen-wins per path keeps true pre-run bytes across multi-turn edits (mirrors pristineRunSnapshot). At run start recoverIfNeeded(repoRoot, writeFileFn, rmFn) replays any surviving in-progress journal — restore each recorded original, delete files the crashed run created (best-effort per entry; failures collected in failed, never thrown) — then deletes it and logs [smallcode] recovered an interrupted run — restored N file(s)…. markClean(repoRoot) (loop.ts, right before return, after the guard) deletes the journal on a clean finish, so recovery is a no-op; a crash skips it, leaving the journal for the next run. Net: edit-apply is atomic at run-granularity across a crash. Both applyBatch (via a journaling writeFileFn wrapper) and the write_file TOOL path route through it. recoverRepo(repoRoot) is a self-contained variant (builds its own traversal-guarded write/rm) for callers that keep the process alive across runs: the smallcode chat REPL calls it in its runLoop-throw catch so a failed task is rolled back to its pre-task state immediately, instead of its journal being replayed (and silently rolling back the START state) by the NEXT REPL task — keeping each REPL task atomic. The one-shot smallcode run path instead process.exit(1)s on a throw, so there the surviving journal is correctly consumed as crash recovery by the next invocation.
tools.tsTool implementations (read_file, write_file, run_command, run_tests, think, finish) and the executeTool dispatcher. All paths guarded by resolveSafe.
bestofn.tsselectBestCandidate — per-TURN candidate selection: generate N completions in parallel, apply each, run verifier, pick winner by verifierScore, revert losers, apply winner. Off by default.
bestofn-loop.tsRun-level oracle-verified Best-of-N. runBestOfNLoop({n, temperatures, setup, verify, deps}) runs the full agent loop up to N independent attempts at the SAME task — fresh env per attempt (setup), a temperature sweep for diversity — and returns the FIRST attempt whose verify (the sound test oracle) goes green, short-circuiting the rest. defaultTemperatures(n) spreads around 1.0 in [0.7, 1.3] (n=1→[1.0]). Returns {passed, attemptsUsed (≤n), winningAttempt, temperatures, states, modelsUsed, winningModelId}. Per-attempt temp injected via the loop's samplingOverride. Wired into the eval runner via SMALLCODE_BEST_OF_N; converts noisy pass@1 into pass@N(any) with zero selection error. R1 model-escalation: optional models: EscalationRung[] makes the MODEL the per-attempt axis — attempt i retargets state.modelId + deps.profile to models[min(i,len-1)], so a run escalates 3b→7b→14b and only pays for the bigger model on the residual the small model can't solve (resolved on first oracle-green, zero selection error).
escalation.tsR1 ladder builder. buildEscalationLadder({spec, registry, provider}) parses a SMALLCODE_ESCALATION spec (comma-separated model ids, cheapest first) into EscalationRung[] ({id, provider, profile}) for runBestOfNLoop({models}). Rungs share the base provider (local models behind one Ollama endpoint, selected per-request by id → stays offline). Unknown id throws via registry.get (clear config error). Empty/unset spec → undefined (plain temp-swept BoN, no behaviour change).
escalate-on-failure.tsR1 single-shot escalate-on-failure — the sibling of the Best-of-N ladder for the common bestOfN === 1 path. runEscalateOnFailure({models, runAttempt, isSolved, reset, log}) runs the cheapest model as a normal single-shot; if isSolved is false, calls reset (revert ONLY the agent's edits — NO clean-tree requirement, the user's work is preserved) and retries with the next bigger model, stopping on the first solve. reset fires BEFORE attempts 2..n only, so a solved attempt's edits are kept and, if the whole ladder fails, the LAST (biggest) model's attempt remains on disk to inspect. Returns {finalState, solvedModelId?, attemptsUsed}. Pure control-flow — all I/O injected, unit-tested with mocks; throws on an empty ladder.
phase-gate.tsP0#2 statewright-style phase-gated tool access — opt-in, OFF by default (SMALLCODE_PHASE_GATE). Pure derivePhase(state, context) → "explore" | "edit" + PHASE_ALLOWED_TOOLS + EXPLORE_REJECT_MESSAGE. "explore" (no context.targetFile AND no state.lockedTargetPath AND no prior successful read_file) advertises only read/inspect tools and hard-rejects write_file/run_command/FILE:/PATCH:; a pinned/locked target is ALWAYS "edit" (first branch — the 1-turn-solve path is never forced through explore). VERIFY folded into edit (documented clean negative). Both prompt.ts (advertised actions) and loop.ts (write rejection) read from here so they can't drift. A/B (edit-reliability, n=5, 2026-07-02): INERT — explore fired 0× (retrieval always pinned a target → always edit phase), so ON ≡ OFF; kept as an opt-in knob, needs low-confidence-retrieval tasks to demonstrate value.
types.tsTypeScript types: AgentState (+ lastFailureSignature?, stallCount?, redraftCount?, verified? — set true only when the tiered oracle confirmed solved before loop exit; lockedTargetPath? — the FIRST confidently-pinned target-lock path, captured once in loop.ts and never overwritten, so per-turn retrieval drift can't relocate the enforcement target; offTargetStreak? — { path, count }, the consecutive same-path counter driving mis-pin retarget; editablePaths?: string[] — the multi-file target set computed once at target-pin time under SMALLCODE_TARGET_SET, primary target + its resolved direct-import neighbors; carouselIndex?/carouselCount?/carouselFocus? — SMALLCODE_SET_CAROUSEL's current index into editablePaths, sweep counter, and the path the prompt should focus on this turn), AgentConfig (optional promptSet?: PromptSet for GEPA seam), Goal, TurnRecord (+ failureSignature?, redrafted?, answerNow?, diagnostic?), ToolCall, ToolResult, ToolName.
strategy.tsrotateStrategy(n: number): string — returns the nth strategy hint (0-indexed) from a 3-item cycle: "use a set/dedup explicitly", "step through each provided example by hand first", "handle edge cases (empty, negative, boundary) before the main case". Used by loop.ts to vary the redraft nudge each cycle.
index.tsRe-exports the public agent API including PromptSet, defaultPromptSet.
target-set.tsMulti-file target set (SMALLCODE_TARGET_SET, default ON since 2026-07-09) — generalizes the single-file target lock into a BOUNDED editable set. computeEditableSet(primaryRel, repoRoot, maxNeighbors=4) returns the pinned primary target plus the source files it directly imports (one hop, resolvable, repo-local, non-test — capped at maxNeighbors for wander-safety), reusing the import extractor from src/verify/import-check.ts. isInEditableSet(path, editableSet) is the multi-path sibling of isOnTargetPath. Consumed by agent/loop.ts (computes the set once at target-pin time, stores it on AgentState.editablePaths) and edit/applier.ts (ApplyBatchOpts.targetPaths supersedes single targetPath) so a genuinely cross-file fix (a function + the helper module it calls) can edit both files while edits outside the neighborhood still hard-reject. See the SMALLCODE_TARGET_SET env-var row. Companion: carousel.ts below (SMALLCODE_SET_CAROUSEL) makes the harness drive ATTENTION across this same editablePaths set when the model itself can't localize across files.
carousel.tsSet-carousel (SMALLCODE_SET_CAROUSEL, default ON since 2026-07-09, requires SMALLCODE_TARGET_SET) — pure state-transition helper advanceCarousel(state, editablePaths, opts). Advances state.carouselIndex to the next file in editablePaths (wrapping), records it as state.carouselFocus (rendered by prompt.ts's "## FOCUS THIS TURN" block), and resets stallCount/redraftCount/lastFailureSignature so the new focus gets a fresh budget. Capped at editablePaths.length * 2 (two full sweeps) via state.carouselCount; no-op when editablePaths.length <= 1. Threshold gating (stallCount >= STALL_LIMIT) lives at the agent/loop.ts call site, not inside the helper — the call site computes carouselActive BEFORE the stall/redraft block and guards the existing redraft trigger with !carouselActive, so in set-mode the carousel fires on the FIRST stall and takes priority over a same-file redraft (which would otherwise reset stallCount and starve the carousel trigger). Attention-only: never touches lockedTargetPath, editablePaths itself, or any enforcement/oracle/revert state.

src/edit/

FilePurpose
parser.tsParse model output into EditBlock[]. Tries PATCH: (function-level) first, then full-file, then search-replace, then JSON. Public: parse(raw).
applier.tsApply EditBlock to disk. Full-file (search='') overwrites the whole file — guarded by truncationReason(original, replace) which rejects likely-truncated re-emits (unbalanced brackets vs a balanced original, >50% line loss, blanking a non-empty file) so a dropped tail never silently corrupts a file. Search-replace tries exact match, then repairBlock fuzzy repair on a miss (records ApplyResult.repair). flattenedPathCandidate(path) + applyBatch redirect a dots-flattened path (src.stats.ts → src/stats.ts) onto the real existing file instead of creating a stray one. patch-function blocks dispatch via a sentinel prefix. Each ApplyResult carries effectivePath (the path actually written, after path-typo rescue) and originalContent (pre-batch on-disk content) — both consumed by loop.ts's revert-on-regression path to restore the correct file even when the model typo'd the path. Anti-fake-green: applyBatch rejects any edit whose path is a test/spec file (isTestFilePath — matches .test./.spec./tests//__tests__/, checked on both the emitted and the flatten-rescued path) with status: "error" + feedback and NO write — the tests are the oracle, so a model editing them to pass can never produce a false "solved". Public: applyBatch, isTestFilePath, isOnTargetPath, OFF_TARGET_EDIT_REJECTED, truncationReason, flattenedPathCandidate, findElisionMarker, ELISION_DETECTED. Elision detection: truncationReason's FIRST check is now findElisionMarker(replace) — a size-independent scan for abbreviation markers (// ..., /* ... */, # ..., "rest of file unchanged", etc.) that rejects with a stable ELISION_DETECTED marker in the error string, ahead of the older bracket-balance/>50%-shrink heuristics — catches abbreviated whole-file re-emits that don't happen to shrink the line count enough to trip the old heuristic alone. Target-lock (drift enforcement): applyBatch(blocks, readFile, writeFile, { targetPath }) — when targetPath is given, any block whose effective (or emitted) path is not the target is rejected with status:"error", a message naming both paths, and the OFF_TARGET_EDIT_REJECTED marker, BEFORE any read/write. isOnTargetPath(path, targetPath) normalizes ./ and backslashes and rescues a dot-flattened typo of EITHER side via flattenedPathCandidate, so a typo'd target path still counts as on-target. opts is optional — omitted, this is a no-op (byte-identical to pre-lock behavior). Driven from loop.ts, which computes the target and fix-mode gating; see the SMALLCODE_TARGET_LOCK env-var row. Multi-file target set: ApplyBatchOpts.targetPaths?: string[] supersedes the single targetPath — when set (via SMALLCODE_TARGET_SET, computed by src/agent/target-set.ts), a block is on-target if its effective path is IN the set (checked via isInEditableSet) rather than equal to one path; targetPath alone remains byte-identical single-file behavior when the set is off.
patch-function.tsFunction-level edits for large files: parsePatchBlocks, applyPatchBlock (anchors on the function's DEFINITION located by NAME via findDefinitionLines — robust to a small model emitting a drifted signature; the synthetic name "default" anchors on the export default function line; falls back to first-replacement-line match only when name lookup finds nothing; brace-depth scan finds the region end), preserveExport (re-adds a leading export/export default the model dropped — a common slip that un-exports the function and breaks importers), chooseEditFormat (PATCH_LINE_THRESHOLD = 80 lines or >8KB → patch eligible; threshold lowered from 300), extractFunctionSource(content, functionName) (reuses locateFunctionRegion to slice out ONLY the target function's current source — used by prompt.ts's forced-recovery template so the model is shown just the function, not the whole file). The harness — not the model — decides the format deterministically from the pinned target file's size and emits it as the ## Edit Target directive. Fail-safe: ambiguous/not-found anchor → error ApplyResult, no write.
repair.tsFuzzy repair for failed search-replace blocks — strategies: exact → whitespace-normalize → per-line-trim → char-similarity fuzzy (≥0.85). repairBlock(block, content) returns the verbatim recovered search text + strategy/confidence. Wired into applier.ts (no longer dead code): a search/replace that misses exact match retries against the repaired text.
types.tsEditBlock, EditFormat (incl. patch-function), ParseResult, ParseError, ApplyResult, ApplyStatus, ApplyBatchResult, RepairResult.
index.tsRe-exports parse and applyBatch.

watchdog.tsWatchdogProvider — wraps a Provider, measures real-generation throughput (completionTokens / wall-time). Tool contracts

The model emits tool calls as TOOL: name {"arg": "value"} lines. The harness parses them with a line-level regex and executes each in order. think and finish are control-flow only; others are side-effecting.

read_file
TOOL: read_file {"path": "src/foo.ts"}

Read a file relative to repoRoot. Returns file contents (truncated at 8,000 chars with [truncated] marker). Returns (file not found) if missing. Path must stay within repoRoot (traversal check).

  • path (string, required) — relative path to file
  • Output fed back as tool result in next turn's prompt history
write_file
TOOL: write_file {"path": "src/foo.ts", "content": "..."}

Write content to a file. Creates intermediate directories. If requireApproval: true, throws ApprovalRequiredError before writing. Path traversal check enforced. Prefer the FILE: edit format — it is parsed before tool calls and is the intended write mechanism.

  • path (string, required)
  • content (string, required)
run_command
TOOL: run_command {"cmd": "bun tsc --noEmit"}

Run a shell command in repoRoot. Command basename must be in allowedCommands (default: bun, tsc, biome, git). Timeout: 30 seconds. Output truncated at 4,000 chars. Combined stdout+stderr returned.

  • cmd (string, required) — space-separated argv
  • Returns: { success: exitCode===0, output, durationMs }
run_tests
TOOL: run_tests {}

Run bun test (or bun test <pattern>) in repoRoot. Timeout: 60 seconds. "Solved" means zero new failures relative to a pre-loop baseline AND ≥1 passing test. Pre-existing failures captured in the baseline are excluded, so unrelated failing tests in a real repo never block early-stop. On fresh single-file benchmark repos (empty baseline) the behaviour is identical to the original whole-suite-green check. This is the deterministic pass-oracle: when outcome is "solved", the loop breaks immediately.

  • pattern (string, optional) — test file glob/path
  • Returns: { success: exitCode===0 && failCount===0, output, durationMs }
think
TOOL: think {"content": "reasoning here..."}

No-op control tool. The harness records it but takes no action. Allows the model to emit explicit reasoning steps outside <think> tags. Always returns success.

finish
TOOL: finish {"summary": "implemented add()"}

Signal goal completion. The harness calls advanceGoal — marks the current goal done and increments currentGoalIndex. If all goals are done, status becomes done. Note: if the oracle returns "solved" (zero new failures + ≥1 pass vs baseline), the loop early-stops before checking for finish — the baseline-relative early-stop oracle takes precedence.

  • summary (string, optional) — brief description of what was done

Edit format spec

Primary: full-file

Emit FILE: <path> on its own line, then a fenced code block with the complete corrected file. The search field is empty — the applier overwrites the whole file. This is what the system prompt instructs and what both qwen2.5-coder and VibeThinker-3B reliably emit.

# Full-file format FILE: src/math.ts ```ts export function add(a: number, b: number): number { return a + b; } ``` TOOL: run_tests {} TOOL: finish {"summary": "implemented add()"}

Rules enforced by the parser: path must not be empty, absolute, or contain ... Leading ./ is normalized. Backslashes are converted to forward slashes.

Large files: patch-function

For files past chooseEditFormat's threshold (PATCH_LINE_THRESHOLD = 80 lines or >8KB; lowered from 300), re-emitting the whole file wastes context and risks the small model dropping unchanged lines. The PATCH: format replaces a single function instead. The harness picks this format deterministically from the pinned target file's line count and tells the model via the ## Edit Target directive — the model never self-assesses file size. The anchor is the first line of the replacement body (typically the signature); the applier finds the one matching line, brace-depth-scans to the closing }, and swaps only that region. If the anchor is missing or ambiguous it returns an error ApplyResult (no write) so the loop can fall back to full-file — it never silently mis-applies.

# patch-function format (large files) PATCH: src/foo.ts FUNCTION: parseConfig ```ts export function parseConfig(raw: string): Config { return JSON.parse(raw); } ```

Fallback 1: search-replace

Classic diff-like format. The file path is on the line immediately before the <<<<<<< SEARCH marker (3+ angle brackets, case-insensitive). Small models (incl. VibeThinker-3B) have trouble with this format; use full-file instead.

src/math.ts <<<<<<< SEARCH function add(a, b) { return a; } ======= function add(a: number, b: number): number { return a + b; } >>>>>>> REPLACE

Fallback 2: JSON

JSON object or array with "type": "edit". Extracted from fences or bare JSON spans. Only activated if the raw output contains "type":"edit".

{"type": "edit", "file": "src/math.ts", "search": "return a;", "replace": "return a + b;"}

Parse priority

Priority Format Trigger Notes
1st patch-function PATCH: marker Checked first, but only matches when a PATCH: block is present. The harness gates this format on the pinned target file's size (>80 lines) via the ## Edit Target directive. Anchored, fail-safe.
2nd Full-file FILE: marker Returns immediately if blocks found and valid. Primary/default format.
3rd Search-replace <<< SEARCH / ===== / >>> REPLACE Fallback. VibeThinker-3B unreliable on this format.
4th JSON "type":"edit" anywhere in output Last resort. Also parses from markdown fences.
