Executive summary

Source: a full read of ai-boost/awesome-harness-engineering (~30 categories, 566-line README) plus 8 followed sources (outlines, statewright, TraceCoder, llvm-autofix, AgentAssay, AgentLens, LaMR context-pruning). Most of the list is enterprise/cloud/frontier-model tooling that doesn't transfer to a fully-offline sub-14B local-model TDD bug-fixer, so it was filtered out. Three items cleared the bar as genuinely new and cheap to test: state-gated tool access (statewright), process-quality scoring on top of binary solved/not-solved (AgentLens's "lucky pass" taxonomy), and constrained decoding for edit-format validity (outlines). Two items were investigated and explicitly rejected: LaMR's AST/CRF retrieval and TraceCoder's HLLM. AgentAssay's behavioral fingerprinting was deprioritized in the original write-up but has since shipped (see P1 #4).

P0 do next
1. Lucky-pass / process-quality scoring
Source: AgentLens, arxiv.org/abs/2605.12925
DONE
What it is
Classify a passing trajectory as Lucky (blind retry, missing verification, disordered explore/implement/verify) vs Solid vs Ideal — a re-read of transcripts already logged, zero new model calls.
Outcome + evidence
Shipped as scripts/classify-pass-quality.ts, an AgentLens-derived static-analysis pass over transcript-store output. Later found and fixed false-positive classifications on real transcripts (per project memory — the classifier itself needed a bugfix pass once run against live data, consistent with the project's "forensic > aggregate" discipline).
2. Statewright's mechanism → 3-phase tool gate
Source: statewright, github.com/statewright/statewright
INERT
What it is
Per-workflow-phase tool-schema restriction (planning → read-only tools; implementation → edit unlocks; verify → no mutating tools) — statewright reported 2/10→10/10 on a 5-task local-model SWE-bench subset. smallcode already partially does this via target-lock; the gap was nothing stopping an edit-tool call before retrieval/planning locks a target.
Outcome + evidence
Built and A/B'd on the edit-reliability + realrepo suites (same protocol as the BoN/line-edit A/Bs). Result: INERT — the explore-tool restriction fired 0× across the suite (smallcode's tool surface was already small enough that the gate never actually constrained anything in practice). Pooled pass@1 moved 0.93→1.00 but that shift is attributed to noise, not the mechanism, since the gate never engaged. Valid negative result, banked per project's "forensic > aggregate" culture — kept shipped since it's harmless, not because it measurably helped.
P1 worth building, more setup
3. Compiler/module-error surfacing
Source: llvm-autofix, arxiv.org/abs/2603.20075 (abstract-only)
ALREADY DONE
What it is
Surfacing structured compiler/error text instead of raw stdout — a scaffolding win that's cheap for a small model to consume.
Outcome + evidence
Status-check read of src/verify/oracle.ts and src/verify/failure-extract.ts found the tier-2 TypeScript typecheck advisory and a "Cannot find module '...'" → errorType: "module-load" shortcut already exist. This directly closes Dogfood gap #1 ("no compile/module-error feedback loop") from project memory — it was already fixed by the time this research pass ran. Residual gap (non-TS toolchains get no advisory signal) is deferred to backlog until a non-TS suite exists to measure it against.
4. AgentAssay-style behavioral fingerprinting
Source: AgentAssay, arxiv.org/abs/2603.02601
DONE
What it is
pass@k CIs answer "did the success rate change." Fingerprinting answers "did the model's behavior change even at equal success rate" — e.g. a change that starts winning via more Best-of-N retries instead of first-shot correctness, same pass@1, worse cost profile.
Outcome + evidence
Shipped in src/improve/fingerprint.ts — fingerprintDiff diffs a behavioral vector (avgTurns, avgTokens, avgAttemptsUsed, repairRate, thinkOnlyRate) against thresholds (REGRESS_ABS_FALLBACK_THRESHOLD=0.15, DRIFT_REL_THRESHOLD=0.3, per-dimension DRIFT_ABS_FLOORS) and returns a regress/drift/stable verdict — orthogonal to, not a replacement for, pass@k CIs. Validated live against a real drift example.
P2 parked / needs a blocking dependency resolved
5. Constrained decoding (grammar-locked edit format)
Source: outlines, github.com/dottxt-ai/outlines
NO-GO
What it is
Regex/CFG/JSON-schema-constrained token sampling — filters the model's next-token distribution so only grammar-valid continuations are ever sampled. Targets the exact malformed-edit-block failure class src/edit/repair.ts already spends effort recovering from.
Outcome + evidence — RESOLVED 2026-07-02
Ollama exposes constrained decoding only via JSON Schema (format/response_format) over HTTP — raw GBNF is not reachable (the grammar-param PR, ollama/ollama#4525, was closed in favor of structured outputs). Plumbing already existed unused: CompletionRequest.responseFormat flows to the HTTP body (src/provider/openai-client.ts:44-73), and a JSON edit-envelope format already exists (src/edit/parser.ts:202-293). Decisive empirical probe (qwen2.5-coder:7b, local, matched multi-line hard cases): JSON-envelope constrained = 0/6 correct+byte-exact vs free-text SEARCH/REPLACE = 5/6 on the identical bugs — forcing code into a JSON string value flattens newlines/whitespace, so search is no longer byte-exact and the edit silently won't apply. Same "added-precision → regression on a small model" pattern as the rejected LINE_EDIT experiment. Verdict: keep free-text SEARCH/REPLACE. Next cheap lever if revisited: instrument the repair-path invocation rate first (no telemetry exists today) before any format change.

Shipped alongside this research pass

Fixes that landed during this cycle but aren't one of the 5 research items above.

Repair-path telemetry
DONE
What it is
No counter existed around repairBlock / non-exact apply strategy in src/edit/applier.ts — the P2 write-up's own "next cheap lever" needed this before any edit-format change could be sized.
Outcome + evidence
src/eval/repair-metrics.ts's summarizeRepairs(turns) now reports { appliedEdits, repaired, repairRate, byStrategy }. Measured baseline on the edit-reliability suite: 0% repair rate — exact-match apply is already succeeding on every turn in that suite, so the repair path isn't currently the bottleneck there (a real, useful negative result, not an absence of instrumentation).
Headless-approval fix (issue #91)
DONE
What it is
requireApproval:true + no TTY (CI, piped, --json, delegation) made prompt() return null — the interactive y/N gate silently defaulted to "N" and rejected every edit in headless runs.
Outcome + evidence
makeInteractiveApprover (src/cli/commands/review.ts) now takes an ApproverOptions with interactive/bypass: when opts.interactive === false, edits apply unconditionally with a one-time stderr notice pointing at smallcode diff/smallcode undo, instead of rejecting every write for reasons unrelated to the edit itself.
Scoped undo (never discard the user's own work)
DONE
What it is
smallcode undo must revert only what the agent itself changed — never a blanket git restore . / git clean -fd, which would discard the user's own uncommitted edits and untracked files too.
Outcome + evidence
src/cli/commands/review.ts records an agent-change manifest (.smallcode/agent-changes.json) as the dirty-set delta captured before the run minus the dirty set after — recordAgentChanges / revertAgentChanges — so undo reverts exactly those paths. Dry-run by default; --yes applies.

Already-have / already-refuted (do not re-propose)

Carried over verbatim from docs/harness-engineering-roadmap.md so this list isn't re-litigated.

  • Elision detection, forced minimal-PATCH, size-gated edit format — already built (src/edit/parser.ts, src/edit/repair.ts, src/edit/patch-function.ts). AST/hash-anchored protocols were already tested via HashPilot and rejected — "3B can't drive it."
  • Line-anchored edits — explicitly tested and REJECTED (regression on both suites); reducing decision surface for a 3B model helps, adding precision-demanding structure doesn't.
  • Model escalation ladder / Best-of-N — already built and CI-confirmed.
  • Compile/module-error feedback — Dogfood gap #1 from project memory; already closed except the non-TS-toolchain residual (see P1 #3 above).
  • pass@k + bootstrap CIs, real-repo suites, forensic-task tooling — already built and already matches or exceeds the rigor bar the awesome-list's Evals & Verification section recommends.
  • LaMR (AST/CRF multi-rubric context pruning) — investigated, not recommended: requires infra smallcode doesn't have; the existing lexical retrieval scorer is already 40/40 at repo scale.
  • GEPA prompt evolution — separately tried and shelved (overfits on a thin held-out suite). Nothing here reopens it.
