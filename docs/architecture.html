System topology

CLI/runner calls the agent loop, which delegates to the provider (Ollama — qwen2.5-coder:3b/7b production, vibethinker-3b legacy), context engine, and edit protocol. Graders and the eval harness sit outside the hot path. Config and Model Registry are cross-cutting — every component reads them at startup.

Agent turn loop

State is persisted to .smallcode/state.json after every turn (atomic rename). The loop terminates when tests are solved (baseline-relative early-stop oracle), all goals are done, or maxTurns is reached (default 15, eval uses 5). A test-baseline is captured before the first turn; "solved" means zero new failures vs baseline AND ≥1 pass — pre-existing unrelated failures never block early-stop.

Honest completion reporting: status = "done" alone does NOT mean the run succeeded — the model may have called TOOL: finish while tests were still failing. AgentState.verified is set to true only when the tiered oracle confirmed outcome === "solved" before the loop exited. The CLI's classifyCompletion(finalState, statePath) helper maps the combination to an honest outcome: success (exit 0) only when status === "done" && verified === true; a ⚠ Finished, but tests are NOT verified passing warning (exit 1) for unverified done; and ✗ Hit max turns without solving / generic error (exit 1) for all other statuses. max_turns status is now also persisted to state.json immediately on turn exhaustion (both the main path and the error-turn path).

Structured failure feedback: on every "failing" oracle verdict, extractFirstFailure parses the first failing assertion (Expected/Received), errorType, and test label into a FailureDiagnostic. This is stored on the turn and rendered as a Failure (fix THIS): header in the next turn's prompt — replacing a blind 600-char raw slice with a structured block the model can directly act on.

Stall detection + redraft: failureSignature() normalises the diagnostic (strips timings, paths, hex addrs) into a stable hash. When STALL_LIMIT=2 consecutive turns produce the same signature, the next turn's prompt is marked as a redraft: Recent History is suppressed (dead-end context cleared) and a rotating strategy hint nudges a fresh approach. After MAX_REDRAFTS=2 the mechanism is disarmed so the loop doesn't cycle forever.

Think-only recovery: a small local model frequently burns its entire generation budget inside <think> and emits no answer — finish_reason=length with an empty answer after the reasoning is stripped. The loop records that turn as failed and sets answerNowNext so the next turn is drafted under an ANSWER NOW prompt: Recent History is suppressed and the model is told to skip thinking and emit the FILE: block / TOOL: call as the first line. Without this, the identical prompt was re-sent and truncated the same way — the dominant failure mode on real repos before the fix.

SKILL injection (GEPA-skills ①): buildSystemPrompt accepts an optional PromptSet.skill string. When non-empty it appends a ## SKILL block after the base system text — a concise repo-agnostic playbook distilled from passing sessions by src/improve/skill-distiller.ts. The injection is additive: absent skill → output is byte-identical to the pre-GEPA baseline. The distiller is offline (no model calls); run via bun scripts/distill-skill.ts.

Edit-format parse pipeline

src/edit/parser.ts tries three formats in order, returning on the first that produces valid blocks. Full-file is primary — it is what the system prompt instructs the model to use and what both qwen2.5-coder and VibeThinker-3B reliably emit.

Deterministic size-gated format: the harness — not the model — chooses the format from the pinned target file's size. buildContext sets bundle.targetFile = { path, lineCount, format, functionName } and buildTurnPrompt emits an explicit ## Edit Target directive: a small file (≤ PATCH_LINE_THRESHOLD = 80 lines) ⇒ emit the COMPLETE file via FILE:; a large file (> 80 lines, with a known function) ⇒ replace ONLY that one function via PATCH:/FUNCTION:. The model no longer self-assesses "is this file large?" (the old advisory ">300 lines" prose had no effect). The PATCH target function is the one that CONTAINS the bug, chosen by pickTargetFunction: a dominant function (≥50% of the file and ≥2× the next) wins query-independently — the common real-repo shape where a default export IS the implementation, including an anonymous export default function extracted as "default" — else an exact query name-match, else body-content relevance (never a weak name-substring match).

Minimal-diff PATCH (default ON, size-gated): re-emitting a whole large function invites over-editing, so for big target functions the harness now asks for a minimal SEARCH/REPLACE diff that changes only the buggy line(s) instead of the complete function. This is governed by SMALLCODE_DIFF_EDIT (default ON; set 0 to revert to whole-function PATCH) and gated by SMALLCODE_DIFF_MIN_FN (default 30 lines) — below that span, whole-function PATCH is kept since small functions don't benefit and exact-match diffs only add fragility. Confirmed +0.17 OVERALL on the edit-reliability suite. The SEARCH text must be copied byte-for-byte from the shown context; the existing fuzzy-repair applier accepts SEARCH/REPLACE and fails safe (no write) on a missing/ambiguous anchor.

Format-consistent not-applied feedback: when a SEARCH/REPLACE edit (DIFF_EDIT mode, gated by functionLineCount ≥ SMALLCODE_DIFF_MIN_FN) fails to apply, the retry feedback instructs the model to re-emit a SEARCH/REPLACE block with the SEARCH text copied byte-for-byte from the shown current file — not "re-emit the complete file", which contradicts the SR directive and causes truncation on large functions. Whole-file and small-function PATCH targets keep their existing not-applied feedback. applyBatch returns each applied result's effectivePath (the path actually written, after path-typo rescue) and originalContent (pre-batch on-disk content); these are also used by the revert-on-regression path to restore the correct file even when the model typo'd the path. Anti-fake-green: applyBatch rejects any edit whose path is a test/spec file (isTestFilePath) with no write and a feedback error — the tests are the oracle, so a model editing them to pass can never forge a "solved" verdict.

Full-file format syntax

The model emits a FILE: marker on its own line, then a fenced code block containing the complete corrected file. The search field is empty, which tells the applier to overwrite the whole file.

# model output
FILE: src/math.ts
```ts
export function add(a: number, b: number): number {
  return a + b;
}
```
TOOL: run_tests {}
TOOL: finish {"summary": "implemented add()"}

Eval / benchmark harness flow

The HumanEval-TS runner (scripts/run-humaneval.ts) fetches problems from the HuggingFace API, sets up an isolated trial directory, runs the full agent loop with stub-only context, and grades with bun test. The test file is on disk but never shown to the model — results reflect general solutions.

The realrepo suite (7 tasks) evaluates the harness on REAL vendored libraries — lukeed/klona (deep-clone, 4 tasks), lukeed/dequal (deep-equal), lukeed/mri (arg parser), all MIT, zero-dep — plus a hand-built two-file tinycalc arithmetic library (realrepo-tinycalc-tokenfield_1) whose fix requires a genuine cross-file edit: the parser reads the wrong token field, and the correct name is defined only by the lexer's token contract in a sibling file. Bugs are real and seeded, graded by authored bun:test oracles. Tasks require genuine multi-file retrieval and correct reasoning about library internals, beyond the synthetic textkit fixtures. Run with SMALLCODE_SUITE=realrepo bun scripts/run-baseline.ts and swap models with SMALLCODE_MODEL=qwen2.5-coder:3b (or :7b). Honest framing: the Qwen coders ace several real-lib bugs (klona) where the reasoning model scored 0/10; the remaining wall is bug-localization on tasks like mri/dequal where the buggy line must be found first.

Env vars: SMALLCODE_HE_LIMIT (default 20), SMALLCODE_HE_OFFSET (default 0), SMALLCODE_HE_K (default 3), SMALLCODE_HE_MAX_TURNS (default 5), SMALLCODE_HE_TIMEOUT_MS (default 600000 — 10 min per trial), SMALLCODE_TARGET_PIN (default on; set 0 to disable target pinning + size-gating for a pre-A baseline A/B).

The eval task runner (src/eval/task-runner.ts) now calls the production walkRepo → buildContext retrieval path instead of an ad-hoc dir-walk, so evals measure what ships and exercise the same target-pinning + size-gate as live runs.

Measuring stick (trustworthy results): a stochastic 3B at temp≈1.0 makes pass@1 over 5 samples swing wildly (a trivial control task observed swinging 1.00→0.40 between runs), so small effects are invisible. The runner now decouples the sample count n (SMALLCODE_EVAL_N) from the reported k (SMALLCODE_REPORT_KS): it runs n trials and reports pass@k for several k via the unbiased estimator (src/eval/metrics.ts), each with a 95% bootstrap confidence interval (src/eval/stats.ts, seeded → reproducible). Two variants differ significantly only when their CIs don't overlap; overlap means raise n (CI width ~1/√n) rather than trust the point estimate. A grader-side infra-error retry (SMALLCODE_GRADER_RETRIES) keeps transient bun test lockfile/EAGAIN failures from being mis-counted as model failures — those trials are excluded from the denominator, never invented as passes. The same denominator also drops empty-generation wedges — a flapping local backend that returns zero tokens for an entire trial (every turn tagged "infra: empty model generation") would otherwise forge a clean 0.00; a task with no surviving trials prints UNMEASURED instead. Each live snapshot in evals/metrics-history.jsonl records n, per-k rates, CIs, think-only-truncation totals, and the sampling overrides in effect, so history rows are self-describing.

Run-level Best-of-N (oracle-verified): set SMALLCODE_BEST_OF_N=N and each trial becomes up to N independent full agent-loop attempts — a FRESH trial dir per attempt, temperatures swept around 1.0 in [0.7, 1.3] (defaultTemperatures(N)) for diversity — resolving on the FIRST attempt whose graders all pass. The deterministic bun test grader is a sound oracle, so any-attempt-green == solved with ZERO selection error: under BoN the reported pass@1 IS the empirical pass@N(any) of the mechanism. The loop (src/agent/bestofn-loop.ts) short-circuits on the first green, so spent attempts ≤ N; the runner records avgAttemptsUsed (mean attempts/trial = cost) alongside the rate. Wired through the same runBestOfNLoop the CLI ships, so the eval measures production behaviour. Distinct from the per-TURN candidate selection (bestofn.ts), which is off by default.

GEPA reflective prompt evolution

GEPA evolves the executor's system prompt instead of the model weights. A STRONG reflection model reads transcripts from tasks the small executor FAILED, diagnoses the single dominant generalizable failure pattern, and rewrites the system prompt to fix it — preserving the FILE:/PATCH: edit format and the TOOL: run_tests/finish protocol. Each candidate is scored on the full train suite (edit-reliability) and kept on a Pareto front; the best PromptSet is written to evals/gepa-best.json for held-out validation against edit-reliability-heldout.

The reflection model call is injected as complete(prompt) => Promise<string>, so unit tests run with no GPU, no LLM, and no network. makeProviderComplete() / reflectConfigFromEnv() build that complete fn from the existing createProvider + model registry — no hand-rolled HTTP. Reflector model + endpoint are env-configured (SMALLCODE_GEPA_REFLECT_MODEL, optional _BASE_URL/_API_KEY/_MAX_TOKENS), falling back to the loaded provider config. The registered reflector profile is qwen2.5-coder:32b (contextWindow 32768, num_ctx 16384, max_tokens 8192, temp 0.6, Apache-2.0) — a strong build-time optimizer, not an executor arm. The runner (scripts/gepa-run.ts) exposes a SMALLCODE_GEPA_DRY_RUN=1 mode (MockMutator + stubbed runTask) for CI smoke, and a search budget via SMALLCODE_GEPA_GENERATIONS (3) / _TRIALS (5) / _POP_CAP (6). Robustness contract: any malformed or failed reflection returns the parent PromptSet unchanged rather than corrupting the candidate — a bad reflection costs a wasted scoring pass, never a regression.

Key design decisions

Why full-file edits? (SEARCH/REPLACE caused the 0.60 → 0.40 regression)

VibeThinker-3B burned its entire context budget agonizing over exact marker syntax (bracket count, the === separator, whitespace). It emitted nothing parseable. Whole-file rewrite is unambiguous: FILE: path + a fenced code block. The parser sees an explicit FILE: marker — no ambiguity vs S/R. Switched in one commit: 0.40 → 0.96.

Why test-oracle early-stop (baseline-relative)?

"Solved" = zero new failures vs a baseline captured before the loop starts, AND ≥1 passing test. This handles repos with pre-existing unrelated failures: those failures appear in the baseline set and are excluded from the "solved" check, so the agent is not penalised for failures it did not introduce. On fresh single-file benchmark repos (no pre-existing failures) the baseline set is empty and behaviour is identical to the original whole-suite-green check. Breaking immediately: (1) locks the pass; (2) stops burning turns; (3) handles "should-not-edit" tasks — if no new failures exist on turn 1, loop exits.

Why temperature ≥0.6? Never 0.2.

VibeThinker-3B was not trained for low-temperature exploration. At temp 0.2 it degenerates into repetitive, low-diversity output that fails to self-correct. Required sampling params: temp 1.0 (minimum 0.6), top_p 0.95, top_k=-1 (requires Modelfile — the OpenAI-compat route silently ignores it). High variance addressed by best-of-N and k=3 eval protocol. qwen2.5-coder uses temp 0.7 — no reasoning-spiral; diversity is handled by top_k=20 and best-of-N.

Why Ollama num_ctx=8192 (not 32K)?

Ollama defaults to num_ctx=2048, which is too small. 32K caused memory pressure and swap on the eval machine, dropping generation speed from ~44 tok/s to ~3 tok/s. 8K is sufficient for task prompt + chain-of-thought + answer and keeps speed high.

Context budget is derived from num_ctx, not contextWindow

The repo-context budget (src/models/context-budget.ts) is effectiveContextWindow − max_tokens, where effectiveContextWindow = ollamaOptions.num_ctx ?? contextWindow. A prior bug budgeted off the nominal contextWindow (65_536) while Ollama actually ran the model at num_ctx=8192 — a 4× overshoot that overflowed the window and returned HTTP 400 after a few turns. Belt-and-suspenders: fitTurnPromptToWindow re-checks every turn and sheds the largest non-pinned context chunk until system + prompt ≤ num_ctx − max_tokens, so even a ballooning failed-edit turn (full-file re-injection into history) cannot overflow.

Why pin the edit target? (size-gated edit + target-file pinning, Option A)

buildContext identifies the edit TARGET — the highest query-scored non-test, non-barrel source file — and emits it as a single PINNED whole-file chunk. Pinned chunks are never windowed (always the complete file, never a ±20-line slice) and never shed by fitTurnPromptToWindow (which now drops only the largest non-pinned chunk). A small model told to "emit the whole file" cannot do so if it only sees a window or the file got dropped to fit the budget. Barrel/re-export files (export *) are skipped as targets — they define no implementation. The harness then size-gates the edit format deterministically (≤80 lines → FILE: whole-file; >80 → PATCH: one function) instead of leaving the model to self-assess file size. Set SMALLCODE_TARGET_PIN=0 to disable pinning + size-gating and measure the clean pre-A baseline on the identical retrieval path.

Security controls

smallcode is designed for local use with untrusted model output. Controls prevent a misbehaving model from escaping the repo sandbox or executing arbitrary commands.

Control Where enforced What it prevents
Command allowlist src/agent/tools.ts Commands checked against sandbox.allowedCommands by exact basename before execution. Default: bun, tsc, biome, git. Rejected calls return an error to the model, not a silent drop.
Path traversal check src/agent/loop.ts (safeResolve), src/agent/tools.ts (resolveSafe) All paths resolved to absolute form and verified to start with repoRoot + path.sep. Any path escaping the root is rejected before read or write.
Approval gate AgentConfig.requireApproval When sandbox.requireApproval: true (default), destructive writes throw ApprovalRequiredError so callers can prompt the user. Disabled in eval/CI environments.
Network access flag src/config/types.ts sandbox.networkAccess: false (default) — policy signal logged for audit. Agent-run commands should not make outbound network requests.
State file isolation src/agent/state.ts Session state written to .smallcode/state.json via atomic rename. No credentials or API keys serialized into state. Config apiKey never leaves the provider layer.
Reasoning log isolation src/reasoning/logger.ts Reasoning traces extracted from <think> tags are logged separately and never fed back into the prompt context, preventing prompt-injection via self-referential reasoning.
