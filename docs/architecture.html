System topology

CLI/runner calls the agent loop, which delegates to the provider (Ollama — qwen2.5-coder:3b/7b production, vibethinker-3b legacy), context engine, and edit protocol. Graders and the eval harness sit outside the hot path. Config and Model Registry are cross-cutting — every component reads them at startup.

Agent turn loop

State is persisted to .smallcode/state.json after every turn (atomic rename). The loop terminates when tests are solved (baseline-relative early-stop oracle), all goals are done, or maxTurns is reached (default 15, eval uses 5). A test-baseline is captured before the first turn; "solved" means zero new failures vs baseline AND ≥1 pass — pre-existing unrelated failures never block early-stop.

Honest completion reporting: status = "done" alone does NOT mean the run succeeded — the model may have called TOOL: finish while tests were still failing. AgentState.verified is set to true only when the tiered oracle confirmed outcome === "solved" before the loop exited. The CLI's classifyCompletion(finalState, statePath) helper maps the combination to an honest outcome: success (exit 0) only when status === "done" && verified === true; a ⚠ Finished, but tests are NOT verified passing warning (exit 1) for unverified done; and ✗ Hit max turns without solving / generic error (exit 1) for all other statuses. max_turns status is now also persisted to state.json immediately on turn exhaustion (both the main path and the error-turn path).

Structured failure feedback: on every "failing" oracle verdict, extractFirstFailure parses the first failing assertion (Expected/Received), errorType, and test label into a FailureDiagnostic. This is stored on the turn and rendered as a Failure (fix THIS): header in the next turn's prompt — replacing a blind 600-char raw slice with a structured block the model can directly act on. R2 externalize-localization (SMALLCODE_LOCALIZE=1, default off): when the failure is a runtime throw whose stack reached a source line, the loop reads a tight window around that exact line and renders a ## BUG LOCATION … ⟵ FAILED HERE block — handing the small model the where it cannot localize itself. Naturally suppressed for value-mismatch failures (their trace points at the test line, not the bug). An upper-bound probe (SMALLCODE_R2_FORCE_LINE=relpath:line, measurement-only, never a shipped default) forces that same window onto a given line for value-mismatch failures too. It was used to probe the mri floor task (bug = one operator on src/index.js:90): an initial n=5 read gave 7b 1/5 with the line handed, but an n=8 re-run scored 0/8 (pooled ≈1/13, within noise of zero) — the 1/5 was an underpowered draw, not a real lift. Neither capacity (32b + line = 0/5) nor a minimal-edit constraint (7b 0/8) moved it either. The mri wall is genuine model incomprehension of a short-circuit idiom (A || (cond) || consume): both 7b and 32b, even pointed at the exact line, rewrite the one-operator fix into &&/ternary forms that break it. Lesson: read floor-task pass rates at n≥8 before claiming a lift; the remaining un-tried lever is harness-side operator-mutation repair, not model-side prompting.

Harness-side operator-mutation repair (SMALLCODE_MUTATION_REPAIR, default on; set =0 to disable): a last-resort pass that fires only when the model loop finishes a fix-mode run (red baseline) UNSOLVED with a locked target file. It enumerates every single operator flip in that file (enumerateComparisonMutations in src/repair/operator-mutation.ts — pure, priority-ordered: comparison first (equality inversions, then boundary off-by-one, then relational inversion), then logical (&&↔||), then arithmetic (+↔-), capped by SMALLCODE_MUTATION_REPAIR_MAX, default 60), runs the real tiered oracle on each candidate, and keeps the first one that goes fully green, reverting every miss. Candidates are now enumerated against the pristine pre-model target file first, then the current on-disk version (deduped, pristine-first ordering) — recovered via pristineTargetContent, which reads the originalContent the revert system already stamps on the first edit to each file in turn history. This matters because a model that fails a wrong-operator bug often mangles the target on the way down (e.g. rewriting a || short-circuit idiom into &&/ternary forms), and a single-operator flip on that wreck can't reach green even though the same flip on the original file can; the current version is still tried as a fallback. The winning candidate's label now names its base, e.g. !== -> === (original). Candidates are then SCOPED to the locked target function's line range via scopeMutationsToRange (state.lockedTargetRange, captured alongside lockedTargetPath from context.targetFile.functionStartLine/functionEndLine when the target function is known) — an operator flip OUTSIDE the bug function that coincidentally greens a weakly-covered test can no longer be selected as a "fix"; falls back to the full whole-file candidate list when the function range is unknown (and scopeMutationsToRange returns a fresh array on that no-op path, so the loop's in-place clear-and-repush can't self-empty it — a latent bug the 2026-07-07 dogfood surfaced). The whole pass is additionally gated to genuine assertion/logic reds: when the baseline failure is a compile/load error (hasLoadError — a missing export/module, syntax error, or unresolved import), which no operator flip can resolve, both operator-mutation and statement repair are skipped rather than churning the oracle over every candidate. It exists because the mri wrong-operator wall above showed that no model-side lever — exact-line localization, 32b escalation, or a minimal-edit prompt — could fix a single-comparison-operator bug; the operator space is small and the oracle is deterministic, so the harness brute-forces it instead of asking the model to. The pass is deterministic and cannot fake-green (it requires a full-green oracle verdict), and it only runs on runs that already failed, so it can never slow down or destabilize a successful one. A/B (7b, no probe): it lifted the mri floor from 0.00 → 0.88 (non-overlapping CIs — significant), took the realrepo suite from 0.90 [.85–.93] to 0.94 [.91–.97] (22-task, 7b, n=10, pooled n=220), and nudged edit-reliability 0.99 → 1.00 with zero regression — which is why it is now on by default.

Read-after-delete repair (SMALLCODE_RAD_HINT default on since v1.7.1 + SMALLCODE_STATEMENT_REPAIR default off): a disjoint-bug-shape sibling of operator-mutation repair, for a statement-ordering mistake that no operator flip can fix. The pure src/repair/read-after-delete.ts mirrors src/repair/operator-mutation.ts (no AST, no deps): detectReadAfterDelete(source) conservatively scans for the X.delete(K); X.set(K, X.get(K)) pattern (which re-inserts undefined) and returns findings + a human hint; repairReadAfterDelete(source) deterministically hoists the read into a __radVal temp before the delete on a single finding. SMALLCODE_RAD_HINT is model-side: when a failing turn leaves that pattern in the attempt, the next prompt carries a ## STATEMENT ORDER BUG section telling the model to read the value into a variable before the delete, and the MODEL reorders and solves — not a harness rescue. SMALLCODE_STATEMENT_REPAIR is harness-side: same last-resort gating as operator-mutation repair (post-loop, unsolved fix-mode run, locked target), it recovers the model's most-recent attempt via the new latestAttemptContent(state, targetPath) loop helper (mirror of pristineTargetContent — needed because the buggy attempt stores undefined, regresses green tests, and is reverted off disk, so the repair must read the attempt, not the pristine disk file), hoists the read, runs the real oracle, keeps the first green, and records a mutationRepair turn so the lucky-pass audit attributes the solve as rescued. A/B on realrepo-lru-recency_1 (7b, n=8, mutation-repair off): RAD_HINT=1 lifted pass@1 0.13 [.00–.38] → 1.00 [1.00–1.00] (1/8 → 8/8), avg turns 5.0 → 2.3, audit modelSolveRate 1.0, rescued 0 — the first MODEL-SIDE lever to move a genuine realrepo floor (contrast mri, where only the harness operator-mutation rescue worked); STATEMENT_REPAIR=1 lifted it to 1.00 [1.00–1.00] (8/8), every solve logged [statement-repair] SOLVED src/index.js via read-after-delete hoist, audit rescuedRate 1.0, model 0. RAD_HINT was promoted to default-on (v1.7.1) after a full 22-task realrepo A/B (7b, n=8) showed the detector fires on the lru-recency pattern only (0 fires across the other 21 tasks / 176 trials) — inert everywhere else; the two other-task swings in that A/B (toposort-cycle 1.00→0.63, csv-quote 0.63→0.13) were confirmed sampling noise by a lever-off control rerun (toposort-cycle redrew 0.88, csv-quote 0.50). STATEMENT_REPAIR stays opt-in (it never fired in the suite once the hint solved lru-recency first).

Stall detection + redraft: failureSignature() normalises the diagnostic (strips timings, paths, hex addrs) into a stable hash. When STALL_LIMIT=2 consecutive turns produce the same signature, the next turn's prompt is marked as a redraft: Recent History is suppressed (dead-end context cleared) and a rotating strategy hint nudges a fresh approach. After MAX_REDRAFTS=2 the mechanism is disarmed so the loop doesn't cycle forever.

Think-only recovery: a small local model frequently burns its entire generation budget inside <think> and emits no answer — finish_reason=length with an empty answer after the reasoning is stripped. The loop records that turn as failed and sets answerNowNext so the next turn is drafted under an ANSWER NOW prompt: Recent History is suppressed and the model is told to skip thinking and emit the FILE: block / TOOL: call as the first line. Without this, the identical prompt was re-sent and truncated the same way — the dominant failure mode on real repos before the fix.

SKILL injection (GEPA-skills ①): buildSystemPrompt accepts an optional PromptSet.skill string. When non-empty it appends a ## SKILL block after the base system text — a concise repo-agnostic playbook distilled from passing sessions by src/improve/skill-distiller.ts. The injection is additive: absent skill → output is byte-identical to the pre-GEPA baseline. The distiller is offline (no model calls); run via bun scripts/distill-skill.ts.

Static import-resolution gate (SMALLCODE_IMPORT_GATE, default ON since 2026-07-05, set =0 to disable — Lever 2, the proactive fix for the dogfood std/strings hallucinated-import failure; promoted after live-fire validation: caught + reverted a real 3B hallucinated slugify import, zero false-fires, with the global-cache resolver fix). A small model invents a module that does not exist, and the only feedback — R4's Cannot find module — arrives only after a full test run, and only when a test imports the edited file, so the model loops re-emitting the same invented import. This gate runs right after an edit lands and before the oracle: it extracts the import specifiers the edit INTRODUCED (extractImportSpecifiers in src/verify/import-check.ts) and resolves each against ground truth — relative paths via Bun.resolveSync from the file's directory, bare packages against package.json deps + node_modules, builtins always OK. Any specifier that does not resolve reverts that file to its pre-edit content and feeds the model a targeted IMPORT ERROR — 'std/strings' does not resolve … this repo's dependencies are: … that NAMES the deps that exist — a crisper, earlier, more actionable signal than a raw module-not-found. Conservative (a declared dep or any Bun-resolvable specifier passes) to keep false-rejects near zero; only NEW imports on an existing file are checked (test/brand-new files skipped). It is the semantic sibling of the syntactic edit-format validation — grounding an edit in the symbols that actually exist, which needs zero model capability to work.

Final-state regression guard (SMALLCODE_FINAL_STATE_GUARD, default ON since 2026-07-05, set =0 to disable — the "never leave the repo worse than you found it" guarantee that dogfooding exposed as missing; promoted after validation: eval-neutral, realrepo subset pooled pass@1 0.90 = baseline). The per-turn revert above catches regressions during the loop, but it cannot roll back a brand-new file the model created (no pre-edit content to restore) and does not re-check the whole-run end state. So a wandering / partial run could exit with more red on disk than it started. This guard runs absolutely last — after the model loop AND every repair pass — and only when the run ended UNSOLVED. It recaptures the full bun test baseline on the final disk state and compares to the run-START baseline (finalStateWorseThanBaseline): if the repo is strictly worse (higher red count OR a test failing now that was green/absent at baseline), it restores every file the agent touched to its pristine pre-model content (pristineRunSnapshot — the whole-run generalization of pristineTargetContent — plus revertFiles) and deletes any brand-new files it created, then re-verifies the restore reached ≤ baseline ([final-state-guard] reverted N file(s) …, recorded on state.finalStateReverted). It is eval-neutral by construction: it fires only on unsolved runs and reverts to the seeded-bug START state, so an unsolved trial stays unsolved (pass/fail unchanged) — it removes broken residue, it never manufactures a pass. Repair-pass throw containment (dogfood #3, 2026-07-08): the guard is only a backstop if the loop actually reaches it. runOperatorMutationRepair and runStatementRepair each call the real bun test oracle per candidate, which can THROW (e.g. a timeout on a large repo) — an unguarded throw escaped runLoop entirely, orphaning a half-tried candidate on disk AND skipping the guard below it (no log, finalStateReverted stayed null). Both repair fns now wrap their candidate loop in try/catch: on any throw they restore the file to the model's pre-repair edit and return null (UNSOLVED) instead of propagating (logged [mutation-repair] … aborted … / [statement-repair] … aborted …), and runLoop additionally wraps both repair-pass blocks in one outer try/catch as defense-in-depth — so the final-state guard is unconditionally reached even if a repair block throws for any other reason. Both repair fns also take an injectable runOracle param (test-only seam, defaults to runTieredOracle) so this contract is covered by a deterministic unit test without a global module mock.

Off-task-drift guard (dogfood #1 blocker, found via 2 live 7b runs: turn 1 lands a wrong edit on the correct target file, then turns 2-8 wander into unrelated files and never touch it again). While the harness has confidently pinned a single edit target (context.targetFile) AND the oracle still reports that target's test failing AND a LATER goal exists to drift onto (currentGoalIndex+1 < goals.length — the planner's own trailing "write tests" / "run tests" sub-goals for a bug-fix task), TOOL: finish no longer calls advanceGoal. Staying on the same goal means the next turn's getContext re-resolves the identical target file, instead of the prompt's "Current Action" and retrieval query sliding onto a sub-goal that has nothing to do with the fix. Single-goal tasks (e.g. each Best-of-N attempt) are unaffected — finish() still ends the run exactly as before. Paired with a per-turn "## STAY ON TARGET" prompt re-anchor (prompt.ts): restates "Edit ONLY <target>…" every turn while red, escalating to a named warning if turnEditedPaths(turn) shows the PRIOR turn wrote to some other file.

Target lock (hard-reject, drift enforcement) — dogfooding found the prompt-level guard above is not enough: a 7b model ignored "## STAY ON TARGET" and edited an unrelated file 7× in one run. When SMALLCODE_TARGET_LOCK is on (default), a confident single context.targetFile is pinned, AND the pre-loop testBaseline was red (fixModeBaseline), the loop enforces it on both write paths: applyBatch(blocks, read, write, { targetPath }) rejects any FILE:/PATCH: block whose effective path isn't the target (status: "error", nothing written — mirrors the existing test-file guard), and the loop's TOOL: write_file executor skips the call before executeTool runs, feeding back the same "Edit REJECTED — this task fixes only <target>…" message as a failed tool result. isOnTargetPath normalizes slashes/./ and rescues dot-flattened path typos bidirectionally (reused from the existing typo-rescue logic) so a mistyped target path still counts as on-target. No targetFile (multi-file/low- confidence retrieval) or baseline all-green → no enforcement, edits flow through unchanged.

Stable run-level lock (drift can't move the target) — the ORIGINAL lockTargetPath was recomputed from context.targetFile on every turn, and context is rebuilt each turn via getContext: once the model edited an off-target file, that file entered recent-history/context and retrieval re-pinned context.targetFile onto it — silently relocating the "lock" onto the drifted file (or to undefined) and rejecting nothing. Live dogfood: an unrelated file was edited 6× with zero rejections. The fix adds AgentState.lockedTargetPath, captured once — the first turn where state.lockedTargetPath === undefined && fixModeBaseline && context.targetFile !== undefined — and persisted via saveState; never overwritten after that. Every turn's enforcement now reads lockTargetPath from this STABLE field instead of the live per-turn context.targetFile, so an off-target edit entering context can no longer relocate what gets enforced. The "## STAY ON TARGET" prompt (prompt.ts) names state.lockedTargetPath ?? context.targetFile.path too, so the prompt and the enforcement always agree on which file is "the target".

Retarget on persistent mis-pin (dogfood follow-up) — a hard lock on a WRONGLY-pinned target is a dead end: dogfooding hit a task to fix effectiveContextWindow in context-budget.ts, but retrieval pinned types.ts (it merely defines the fields — contextWindow, num_ctx — the query mentioned). The model correctly tried the real file every turn and was rejected all 4 turns to max_turns — unsolvable regardless of model size. The fix distinguishes random drift (a different off-target file each turn — keep rejecting) from a persistent mis-pin (the SAME off-target file, turn after turn — retarget). loop.ts's trackOffTargetAttempt tracks AgentState.offTargetStreak: { path, count }, a consecutive-same-path counter over rejected attempts; when the streak reaches OFF_TARGET_RETARGET_THRESHOLD (2), the lock gives up on the stale pin, sets state.lockedTargetPath to the persistently-attempted path, and lets that same turn's write through — so a mis-pin self-corrects within one extra turn instead of dead-locking to max_turns. An on-target attempt or a switch to a DIFFERENT off-target path resets the streak, so genuine drift is rejected forever exactly as before; test files (isTestFilePath) are never a retarget candidate — the anti-fake-green guard is untouched. Both write paths (applyBatch and the TOOL: write_file guard) share one per-turn lockTargetPathForTurn, so a retarget triggered by one write path also applies to the other write inside the same turn.

Edit-format parse pipeline

src/edit/parser.ts tries three formats in order, returning on the first that produces valid blocks. Full-file is primary — it is what the system prompt instructs the model to use and what both qwen2.5-coder and VibeThinker-3B reliably emit.

Deterministic size-gated format: the harness — not the model — chooses the format from the pinned target file's size. buildContext sets bundle.targetFile = { path, lineCount, format, functionName } and buildTurnPrompt emits an explicit ## Edit Target directive: a small file (≤ PATCH_LINE_THRESHOLD = 80 lines) ⇒ emit the COMPLETE file via FILE:; a large file (> 80 lines, with a known function) ⇒ replace ONLY that one function via PATCH:/FUNCTION:. The model no longer self-assesses "is this file large?" (the old advisory ">300 lines" prose had no effect). The PATCH target function is the one that CONTAINS the bug, chosen by pickTargetFunction: a dominant function (≥50% of the file and ≥2× the next) wins query-independently — the common real-repo shape where a default export IS the implementation, including an anonymous export default function extracted as "default" — else an exact query name-match, else body-content relevance (never a weak name-substring match).

Minimal-diff PATCH (default ON, size-gated): re-emitting a whole large function invites over-editing, so for big target functions the harness now asks for a minimal SEARCH/REPLACE diff that changes only the buggy line(s) instead of the complete function. This is governed by SMALLCODE_DIFF_EDIT (default ON; set 0 to revert to whole-function PATCH) and gated by SMALLCODE_DIFF_MIN_FN (default 30 lines) — below that span, whole-function PATCH is kept since small functions don't benefit and exact-match diffs only add fragility. Confirmed +0.17 OVERALL on the edit-reliability suite. The SEARCH text must be copied byte-for-byte from the shown context; the existing fuzzy-repair applier accepts SEARCH/REPLACE and fails safe (no write) on a missing/ambiguous anchor.

Format-consistent not-applied feedback: when a SEARCH/REPLACE edit (DIFF_EDIT mode, gated by functionLineCount ≥ SMALLCODE_DIFF_MIN_FN) fails to apply, the retry feedback instructs the model to re-emit a SEARCH/REPLACE block with the SEARCH text copied byte-for-byte from the shown current file — not "re-emit the complete file", which contradicts the SR directive and causes truncation on large functions. Whole-file and small-function PATCH targets keep their existing not-applied feedback. applyBatch returns each applied result's effectivePath (the path actually written, after path-typo rescue) and originalContent (pre-batch on-disk content); these are also used by the revert-on-regression path to restore the correct file even when the model typo'd the path. Anti-fake-green: applyBatch rejects any edit whose path is a test/spec file (isTestFilePath) with no write and a feedback error — the tests are the oracle, so a model editing them to pass can never forge a "solved" verdict.

Elision-detection + forced-PATCH recovery: the #1 real-repo usability blocker was a PATCH-target model instead re-emitting an abbreviated whole file (// ... / # ... / "rest of file unchanged" placeholders standing in for elided code). The truncation guard (truncationReason in src/edit/applier.ts) now runs findElisionMarker as its first, size-independent check and rejects with a stable ELISION_DETECTED marker in the error — before, only the >50%-line-shrink heuristic caught this, which missed abbreviations that still looked bulky. Rejecting alone was not enough: the OLD not-applied feedback re-showed the whole file and said "don't emit the whole file", which is the exact pattern that produced the mistake, so a small model repeated it every turn and never landed an edit (max_turns, zero progress — no false-green, but no progress either). buildTurnPrompt now detects this specific shape — editBlocks[i].search === "" or ELISION_DETECTED in the apply error, for a PATCH-format target — ahead of the generic SR/PATCH retry branches, and forces a different remedy: extractFunctionSource (reusing locateFunctionRegion) isolates ONLY the target function's current text (never the whole file) and the feedback emits a literal copy-pasteable <<<<<<< SEARCH / ======= / >>>>>>> REPLACE template with explicit "Do NOT re-emit the file. Do NOT use // ...." wording, escalating to sterner "FINAL chance" wording if the same mismatch repeats in the very next turn (wholeFileMismatchSeen counter over the last 2 turns). Default-on, no env gate; additive — falls through to the existing SR/PATCH/FILE retry branches when the mismatch shape isn't detected. Deterministic prompt-level regression test: tests/prompt-recovery-elision.test.ts.

Measuring stick (trustworthy results): a stochastic 3B at temp≈1.0 makes pass@1 over 5 samples swing wildly (a trivial control task observed swinging 1.00→0.40 between runs), so small effects are invisible. The runner now decouples the sample count n (SMALLCODE_EVAL_N) from the reported k (SMALLCODE_REPORT_KS): it runs n trials and reports pass@k for several k via the unbiased estimator (src/eval/metrics.ts), each with a 95% bootstrap confidence interval (src/eval/stats.ts, seeded → reproducible). Two variants differ significantly only when their CIs don't overlap; overlap means raise n (CI width ~1/√n) rather than trust the point estimate. A grader-side infra-error retry (SMALLCODE_GRADER_RETRIES) keeps transient bun test lockfile/EAGAIN failures from being mis-counted as model failures — those trials are excluded from the denominator, never invented as passes. The same denominator also drops empty-generation wedges — a flapping local backend that returns zero tokens for an entire trial (every turn tagged "infra: empty model generation") would otherwise forge a clean 0.00; a task with no surviving trials prints UNMEASURED instead. Each live snapshot in evals/metrics-history.jsonl records n, per-k rates, CIs, think-only-truncation totals, and the sampling overrides in effect, so history rows are self-describing.

Run-level Best-of-N (oracle-verified): set SMALLCODE_BEST_OF_N=N and each trial becomes up to N independent full agent-loop attempts — a FRESH trial dir per attempt, temperatures swept around 1.0 in [0.7, 1.3] (defaultTemperatures(N)) for diversity — resolving on the FIRST attempt whose graders all pass. The deterministic bun test grader is a sound oracle, so any-attempt-green == solved with ZERO selection error: under BoN the reported pass@1 IS the empirical pass@N(any) of the mechanism. The loop (src/agent/bestofn-loop.ts) short-circuits on the first green, so spent attempts ≤ N; the runner records avgAttemptsUsed (mean attempts/trial = cost) alongside the rate. Wired through the same runBestOfNLoop the CLI ships, so the eval measures production behaviour. Distinct from the per-TURN candidate selection (bestofn.ts), which is off by default. R1 model-escalation ladder (SMALLCODE_ESCALATION, default off): a comma-separated model list (cheapest first, e.g. 3b,3b,7b) makes the MODEL the per-attempt axis — attempt i runs with rung min(i, len-1). Because BoN resolves on the first oracle-green attempt, a run only pays for a bigger model on the residual the small one can't solve (zero selection error). All rungs share the local Ollama endpoint, so escalation stays fully offline — cap at the largest LOCAL model. The winning rung is recorded per trial. Honest framing: single-shot's own iid pass@3 estimate already ≈ 1.00, so BoN doesn't beat the diversity ceiling — it OPERATIONALIZES it into a delivered, oracle-verified green.

Measured on the harder realrepo suite (2026-06-28, qwen2.5-coder:3b, n=10/task × 7): pooled pass@1 0.66 [.54-.76] → BoN=3 0.76 [.66-.86] — CIs OVERLAP, ns at n=10. Takeaway: BoN is a diversity amplifier, not a capability adder — its payoff scales with how much un-delivered iid headroom a suite actually has.

GEPA reflective prompt evolution

GEPA evolves the executor's system prompt instead of the model weights. A STRONG reflection model reads transcripts from tasks the small executor FAILED, diagnoses the single dominant generalizable failure pattern, and rewrites the system prompt to fix it — preserving the FILE:/PATCH: edit format and the TOOL: run_tests/finish protocol. Each candidate is scored on the full train suite (edit-reliability) and kept on a Pareto front; the best PromptSet is written to evals/gepa-best.json for held-out validation against edit-reliability-heldout.

Key design decisions

Why full-file edits? (SEARCH/REPLACE caused the 0.60 → 0.40 regression)

VibeThinker-3B burned its entire context budget agonizing over exact marker syntax (bracket count, the === separator, whitespace). It emitted nothing parseable. Whole-file rewrite is unambiguous: FILE: path + a fenced code block. The parser sees an explicit FILE: marker — no ambiguity vs S/R. Switched in one commit: 0.40 → 0.96.

Why test-oracle early-stop (baseline-relative)?

"Solved" = zero new failures vs a baseline captured before the loop starts, AND ≥1 passing test. This handles repos with pre-existing unrelated failures: those failures appear in the baseline set and are excluded from the "solved" check, so the agent is not penalised for failures it did not introduce. On fresh single-file benchmark repos (no pre-existing failures) the baseline set is empty and behaviour is identical to the original whole-suite-green check. Breaking immediately: (1) locks the pass; (2) stops burning turns; (3) handles "should-not-edit" tasks — if no new failures exist on turn 1, loop exits.

Why temperature ≥0.6? Never 0.2.

VibeThinker-3B was not trained for low-temperature exploration. At temp 0.2 it degenerates into repetitive, low-diversity output that fails to self-correct. Required sampling params: temp 1.0 (minimum 0.6), top_p 0.95, top_k=-1 (requires Modelfile — the OpenAI-compat route silently ignores it). High variance addressed by best-of-N and k=3 eval protocol. qwen2.5-coder uses temp 0.7 — no reasoning-spiral; diversity is handled by top_k=20 and best-of-N.

Why does vibethinker-3b stay at num_ctx=8192 (not 32K)?

Ollama defaults to num_ctx=2048, which is too small. On the constrained eval machine vibethinker-3b was profiled on, 32K caused memory pressure and swap, dropping generation speed from ~44 tok/s to ~3 tok/s, so its profile keeps num_ctx=8192 (sufficient for task prompt + chain-of-thought + answer on that box). The production qwen2.5-coder:3b and qwen2.5-coder:7b profiles were raised to their native num_ctx=32768 (2026-07-08) — that eval-machine caveat doesn't apply on modern hardware with ample unified memory (e.g. Apple M5 Max / 48GB), and the old 8K value left only ~4K of usable prompt budget (num_ctx − max_tokens), starving repo-context localization on large repos.

Context budget is derived from num_ctx, not contextWindow

The repo-context budget (src/models/context-budget.ts) is effectiveContextWindow − max_tokens, where effectiveContextWindow = ollamaOptions.num_ctx ?? contextWindow. A prior bug budgeted off the nominal contextWindow (65_536) while Ollama actually ran the model at num_ctx=8192 — a 4× overshoot that overflowed the window and returned HTTP 400 after a few turns. Belt-and-suspenders: fitTurnPromptToWindow re-checks every turn and sheds the largest non-pinned context chunk until system + prompt ≤ num_ctx − max_tokens, so even a ballooning failed-edit turn (full-file re-injection into history) cannot overflow.

Why pin the edit target? (size-gated edit + target-file pinning, Option A)

buildContext identifies the edit TARGET — the highest query-scored non-test, non-barrel source file — and emits it as a single PINNED whole-file chunk. Pinned chunks are never windowed (always the complete file, never a ±20-line slice) and never shed by fitTurnPromptToWindow (which now drops only the largest non-pinned chunk). A small model told to "emit the whole file" cannot do so if it only sees a window or the file got dropped to fit the budget. Barrel/re-export files (export *) are skipped as targets — they define no implementation. The harness then size-gates the edit format deterministically (≤80 lines → FILE: whole-file; >80 → PATCH: one function) instead of leaving the model to self-assess file size. Set SMALLCODE_TARGET_PIN=0 to disable pinning + size-gating and measure the clean pre-A baseline on the identical retrieval path.

Improve src/improve/ Session logging, candidate task promotion from transcripts, metrics history (evals/metrics-history.jsonl), regression gating, A/B prompt comparison. SessionLogger, MetricsStore, RegressionCheckResult, ABResult, runGate

Security controls

smallcode is designed for local use with untrusted model output. Controls prevent a misbehaving model from escaping the repo sandbox or executing arbitrary commands.

Control Where enforced What it prevents
Command allowlist src/agent/tools.ts Commands checked against sandbox.allowedCommands by exact basename before execution. Default: bun, tsc, biome, git. Rejected calls return an error to the model, not a silent drop.
Path traversal check src/agent/loop.ts (safeResolve), src/agent/tools.ts (resolveSafe) All paths resolved to absolute form and verified to start with repoRoot + path.sep. Any path escaping the root is rejected before read or write.
Approval gate AgentConfig.requireApproval When sandbox.requireApproval: true (default), destructive writes throw ApprovalRequiredError so callers can prompt the user. Disabled in eval/CI environments.
Network access flag src/config/types.ts sandbox.networkAccess: false (default) — policy signal logged for audit. Agent-run commands should not make outbound network requests.
State file isolation src/agent/state.ts Session state written to .smallcode/state.json via atomic rename. No credentials or API keys serialized into state. Config apiKey never leaves the provider layer.
Reasoning log isolation src/reasoning/logger.ts Reasoning traces extracted from <think> tags are logged separately and never fed back into the prompt context, preventing prompt-injection via self-referential reasoning.
